### M5 — Soroban treasury contract written (cargo test 4/4 PASS)
**Done:** Authored the trust core, `contracts/treasury/src/lib.rs` — one contract instance = one
pool; holds USDC as a Stellar Asset Contract (SAC); releases funds only when the on-chain policy is
satisfied. Workspace pins `soroban-sdk = "26"` with a size-optimized release profile
(`opt-level = "z"`, `lto = true`, `panic = "abort"`, `codegen-units = 1`, `strip = "symbols"`,
`overflow-checks = true`).

Storage keys (`DataKey`): `Token`, `Officers`, `Threshold`, `NextSpendId`, `Members`, `Categories`,
`CategoryLimit(Symbol)`, `Contribution(Address)`, `Spend(u32)`.
Structs: `CategoryInfo { name: Symbol, limit: i128 }`,
`SpendRequest { id, proposer, category, amount, recipient, memo, approvals: Vec<Address>, executed: bool }`.

**Exact function signatures (as implemented):**

```rust
pub fn initialize(env: Env, token: Address, officers: Vec<Address>, threshold: u32,
                  categories: Vec<Symbol>, limits: Vec<i128>)          // one-time; validates threshold + lengths
pub fn contribute(env: Env, from: Address, amount: i128)              // from.require_auth(); transfers in; records member
pub fn request_spend(env: Env, proposer: Address, category: Symbol,   // proposer.require_auth(); must be officer;
                     amount: i128, recipient: Address, memo: String)  //   checks category cap; auto-approves proposer
                     -> u32
pub fn approve(env: Env, officer: Address, spend_id: u32)             // officer.require_auth(); must be officer; no dup
pub fn execute(env: Env, spend_id: u32)                              // PERMISSIONLESS; reverts unless approvals >= threshold
// views:
pub fn get_balance(env) -> i128            pub fn get_spend(env, id) -> Option<SpendRequest>
pub fn get_spends(env) -> Vec<SpendRequest> pub fn get_categories(env) -> Vec<CategoryInfo>
pub fn get_officers(env) -> Vec<Address>   pub fn get_threshold(env) -> u32
pub fn get_members(env) -> Vec<Address>    pub fn get_contribution(env, who) -> i128
pub fn get_next_spend_id(env) -> u32
```

**Error codes** (`#[contracterror] #[repr(u32)]`):

| # | Error | Guards |
|---|---|---|
| 1 | `AlreadyInitialized` | second `initialize` |
| 2 | `NotOfficer` | non-officer proposes/approves |
| 3 | `OverCategoryLimit` | spend > category cap (cap `> 0`) |
| 4 | `SpendNotFound` | unknown spend id |
| 5 | `AlreadyExecuted` | approve/execute an executed spend |
| 6 | `NotEnoughApprovals` | `execute` below threshold |
| 7 | `AlreadyApproved` | officer approves twice |
| 8 | `InvalidInit` | bad threshold / mismatched lengths |
| 9 | `InsufficientBalance` | treasury can't cover the transfer |
| 10 | `NonPositiveAmount` | `contribute`/`request_spend` amount `<= 0` |

Events published via `env.events().publish`: `contrib`, `spend_req`, `approve`, `execute`.

**Tests (`contracts/treasury/src/test.rs`) — 4/4 PASS:**

| Test | Asserts |
|---|---|
| `full_happy_path` | contribute 2,000 → request 1,200 Equipment (auto-approve) → officer1 approve → execute → vendor gets 1,200, treasury 800, `executed == true` |
| `cannot_execute_without_threshold` | `try_execute` with 1 approval → `Err(NotEnoughApprovals #6)`, vendor still 0 |
| `rejects_over_category_limit` | `try_request_spend` 6,000 vs Equipment cap 5,000 → `Err(OverCategoryLimit #3)` |
| `non_officer_cannot_propose` | plain member proposes → `Err(NotOfficer #2)` |

The test harness registers a v2 SAC (`register_stellar_asset_contract_v2`), 3 generated officers,
a member and a vendor, and initializes 2-of-3 with categories Equipment/Venue (caps 5,000 / 3,000).

**Outcome:** The moat — "one officer cannot drain, non-officers cannot propose, over-limit is
rejected" — is proven in Rust before any deployment.

## Problems encountered and exactly how each was solved

### 1. `ed25519-dalek 3.0.0` broke the test build
**Symptom:** A freshly released `ed25519-dalek 3.0.0` broke `soroban-env-host` testutils; contract
tests would not compile.
**Fix:** Pinned it back with `cargo update -p ed25519-dalek --precise 2.2.0`. This affects the **test
toolchain only** — the deployable wasm build was unaffected.

### 2. Test assertions couldn't match a `panic_with_error`
**Symptom:** The contract signals policy violations via `panic_with_error!`, but a plain `#[should_panic]`
couldn't assert the *specific* error code.
**Fix:** Used the generated `try_` client (`try_execute`, `try_request_spend`), which surfaces the
panic as a `soroban_sdk::Error`, and asserted exact codes with
`Err(Ok(soroban_sdk::Error::from_contract_error(Error::NotEnoughApprovals as u32)))`. This is how all
three negative tests (#6, #3, #2) pin their errors.

### 3. Browser needs Node globals for `@stellar/stellar-sdk`
**Symptom:** `stellar-sdk` expects `Buffer` / `global` / `process`, which don't exist in the browser,
breaking the web build/runtime.
**Fix:** Added `vite-plugin-node-polyfills` (`^0.28.0`) to provide Buffer/global/process shims.

### 4. Generated bindings vs. strict TypeScript config
**Symptom:** The generated bindings didn't satisfy the app's strict tsconfig
(`verbatimModuleSyntax`), and lacked Node types.
**Fix:** Turned `verbatimModuleSyntax` **off** and added `@types/node` (`^26.1.1`) so the generated
`Client` compiles cleanly.

### 5. Backend env leak forced the CLI to demand a passphrase
**Symptom:** A `SOROBAN_RPC_URL` picked up from `.env` leaked into the Stellar CLI child process; the
CLI then required a matching network passphrase and the invoke failed.
**Fix:** In the `stellar()` helper, clone `process.env` and **delete** `SOROBAN_RPC_URL`,
`STELLAR_RPC_URL`, `STELLAR_NETWORK_PASSPHRASE`, `SOROBAN_NETWORK_PASSPHRASE` before `execFile`, so
`--network testnet` fully defines the network.

### 6. Classic-asset recipients need a trustline first
**Symptom:** Minting USDC **into the contract** worked (contracts are trustline-exempt), but the first
`transfer` to a fresh classic account failed until that account trusted USDC.
**Fix:** Onboarding creates trustlines client-side: `ensureTrustline(kp, issuer)` builds a
`change_trust` op to `USDC:<issuer>` for each persona **and** the payee before any disbursement. A real
SEP-24 anchor would do this automatically — which is exactly the composability story.

### 7. Units — everything on-chain is raw token units
**Symptom:** USDC SAC has **7 decimals**; the contract stores unit-agnostic `i128`. Mixing "human"
and "raw" amounts would misprice everything.
**Fix:** Fixed one convention app-wide: `SCALE = 10_000_000`, `raw = human × 1e7`
(`rawToUsd` / `usdToRaw`). Demo treats **1 USDC ≈ ₱1**. So dues ₱200 → `2_000_000_000`; the 20,000
seed → `200_000_000_000`; Equipment cap ₱5,000 → `50_000_000_000`. The frontend formats raw → ₱ for
display and multiplies by SCALE for any contract call.

### 8. Dev-process port lingered after stopping a background task
**Symptom:** Stopping the background dev task didn't always kill its child process; the port stayed
bound, so a restart failed.
**Fix:** Kill the port's listener before restarting the dev server.

# Backend & AI Service
_Created: 2026-07-11 01:46 MPST · Kolektibo · Stellar Testnet_

Reference for **`services/ai`** — the single Node/Express process that wears two hats for Kolektibo:

1. **The AI treasurer.** It turns a group's plain-language money rules into a strict, on-chain-ready JSON policy (`/rules`) and answers members' questions grounded in live on-chain state (`/ask`), using OpenAI.
2. **A thin chain-ops backend.** It performs the two privileged operations the browser cannot do safely — minting test USDC (`/faucet`) and deploying + initializing a fresh treasury contract (`/pool/create`) — by shelling out to the Stellar CLI, which holds the issuer/deployer identities in its local keystore.

Everything trust-critical about the money lives in the Soroban contract, not here. This service is **stateless** — it has no database and holds no authority over funds. Its worst-case power is minting *test* USDC and deploying *new* pools; it can never move money out of an existing treasury (only the contract's on-chain multisig policy can).

Source of truth for this document: `services/ai/src/index.ts`, `services/ai/package.json`, `services/ai/.env.example`, and the consuming frontend modules `apps/web/src/lib/ai.ts` and `apps/web/src/lib/backend.ts`.

## 2. Server bootstrap

```ts
import 'dotenv/config'
import express from 'express'
import cors from 'cors'
import OpenAI from 'openai'
import { z } from 'zod'
import { execFile } from 'node:child_process'
import { promisify } from 'node:util'

const pExecFile = promisify(execFile)

const app = express()
app.use(cors())                       // open CORS — browser hits this directly
app.use(express.json({ limit: '1mb' }))

const apiKey = process.env.OPENAI_API_KEY
const model  = process.env.OPENAI_MODEL || 'gpt-4o-mini'
const client = apiKey ? new OpenAI({ apiKey }) : null   // null when no key
```

Key behaviors:

- **CORS is fully open** (`app.use(cors())`). The frontend at `http://localhost:5173` (Vite) calls the service at `http://localhost:8787` directly.
- **JSON body limit is `1mb`.** This matters for `/ask`, which receives the full on-chain state blob in the request body.
- **The OpenAI client is created lazily.** If `OPENAI_API_KEY` is unset, `client` is `null`, and the two AI endpoints (`/rules`, `/ask`) return **HTTP 500** with `OPENAI_API_KEY not set on the AI service`. The chain-ops endpoints (`/config`, `/faucet`, `/pool/create`) do **not** require an OpenAI key and keep working.

### Startup

```ts
const port = Number(process.env.PORT || 8787)
app.listen(port, () => {
  console.log(`Kolektibo AI service → http://localhost:${port}  (model: ${model}, key: ${apiKey ? 'set' : 'MISSING'})`)
})
```

The boot log tells you at a glance whether the OpenAI key was picked up (`key: set` vs `key: MISSING`).

### 4.2 `POST /rules`

Converts a group's plain-language money rules into a strict JSON policy that maps onto how the contract is initialized. Uses OpenAI in `json_object` mode at **temperature 0** (deterministic), then validates the model's output with a **zod** schema before returning it.

**Request body**

```json
{ "text": "Everyone pays ₱200 a month. Spends over ₱5,000 need 2 of 3 officers to approve." }
```

| Field | Type | Rules |
| --- | --- | --- |
| `text` | `string` | Required, trimmed. Empty → **400** `Missing "text"`. |

**Response `200`** — the validated policy:

```json
{
  "policy": {
    "currency": "USDC",
    "dues": { "amount": 200, "period": "monthly" },
    "categories": [
      { "name": "Equipment", "monthlyLimit": 5000 },
      { "name": "Venue", "monthlyLimit": 3000 }
    ],
    "approval": { "threshold": 2, "of": 3 },
    "summary": "Everyone chips in ₱200 a month, and any spend over ₱5,000 needs 2 of your 3 officers to approve."
  }
}
```

**Policy shape (zod `PolicySchema`)** — this is enforced server-side; anything the model returns that doesn't fit is rejected:

| Field | Zod type | Notes |
| --- | --- | --- |
| `currency` | `z.string().default('USDC')` | Defaults to `USDC` if omitted. |
| `dues` | `{ amount: number, period: 'monthly'\|'weekly'\|'once' } \| null` | `amount` uses `z.coerce.number()`; `null` when no dues stated. |
| `categories` | `Array<{ name: string, monthlyLimit: number \| null }>` | `monthlyLimit` coerced number or `null` (no cap). |
| `approval` | `{ threshold: int ≥ 1, of: int ≥ 1 }` | Both `z.coerce.number().int().min(1)`. |
| `summary` | `z.string()` | One warm, plain-language sentence. |

The matching TypeScript type on the client (`apps/web/src/lib/ai.ts`):

```ts
export type Policy = {
  currency: string
  dues: { amount: number; period: 'monthly' | 'weekly' | 'once' } | null
  categories: { name: string; monthlyLimit: number | null }[]
  approval: { threshold: number; of: number }
  summary: string
}
```

**Errors**

| Status | Body | Cause |
| --- | --- | --- |
| `400` | `Missing "text"` | Empty/absent `text`. |
| `422` | zod / JSON error message | Model returned JSON that fails `PolicySchema.parse` (or non-JSON). |
| `500` | `OPENAI_API_KEY not set on the AI service` | No OpenAI key. |

**Implementation**

```ts
const completion = await client.chat.completions.create({
  model,
  temperature: 0,
  response_format: { type: 'json_object' },
  messages: [
    { role: 'system', content: RULES_SYSTEM },
    { role: 'user', content: text },
  ],
})
const raw = completion.choices[0]?.message?.content ?? '{}'
const policy = PolicySchema.parse(JSON.parse(raw))  // 422 on failure
res.json({ policy })
```

### 4.3 `POST /ask`

Answers a member's question in warm, plain language, **grounded only in the on-chain state the caller passes in** (balance, contributions, spend history, policy). The AI never fetches chain data itself and is instructed to never invent numbers. Runs at **temperature 0.2** (slightly warmer than `/rules`), no `json_object` mode — the answer is free text.

**Request body**

```json
{
  "question": "How much do we have and where did it go?",
  "state": {
    "balance": 300,
    "contributions": [ { "who": "Kap. Ramon", "amount": 200 } ],
    "spends": [ { "category": "Equipment", "amount": 300, "recipient": "…", "executed": true } ],
    "policy": { "threshold": 2, "of": 3 }
  }
}
```

| Field | Type | Rules |
| --- | --- | --- |
| `question` | `string` | Required, trimmed. Empty → **400** `Missing "question"`. |
| `state` | `object` (any JSON) | Optional; defaults to `{}`. Serialized into the prompt verbatim. |

**Response `200`**

```json
{ "answer": "You've got ₱300 in the fund right now. The last spend was ₱300 for Equipment, which the officers approved and released. Everything checks out on-chain." }
```

**Errors**

| Status | Body | Cause |
| --- | --- | --- |
| `400` | `Missing "question"` | Empty/absent `question`. |
| `500` | `OPENAI_API_KEY not set on the AI service` | No OpenAI key. |
| `500` | error message | OpenAI call failed. |

**Prompt construction** — the state is embedded as pretty-printed JSON in the user message:

```ts
messages: [
  { role: 'system', content: ASK_SYSTEM },
  { role: 'user', content:
      `On-chain state (JSON):\n${JSON.stringify(state, null, 2)}\n\nQuestion: ${question}` },
]
```

Returns the **public** chain configuration the browser needs to talk to Stellar directly (RPC, network passphrase, USDC asset, category policy). Contains **no secrets** — every value here is public or a client-side constant.

**Response `200`**

```json
{
  "network": "testnet",
  "usdcSac": "CDTCIZLKSZNDFDSZRQUFIHQ5P5L2OOI5DDOMSY5NH6NQQTGSOE5LK7QR",
  "usdcIssuer": "GBYFIFSFQUE6M4O4ESBX7I4FU2XXPRI3V47C2BONMZBG6VKYCBSG55HM",
  "rpcUrl": "https://soroban-testnet.stellar.org",
  "passphrase": "Test SDF Network ; September 2015",
  "friendbotUrl": "https://friendbot.stellar.org",
  "categories": ["Equipment", "Venue", "Refreshments"],
  "limits": ["50000000000", "30000000000", "15000000000"],
  "threshold": 2,
  "configured": true
}
```

| Field | Type | Source / meaning |
| --- | --- | --- |
| `network` | `string` | `STELLAR_NETWORK` (default `testnet`). |
| `usdcSac` | `string` | `USDC_SAC_ID` — the test USDC Stellar Asset Contract. |
| `usdcIssuer` | `string` | `USDC_ISSUER` — classic issuer public key. |
| `rpcUrl` | `string` | `SOROBAN_RPC_URL` (default `https://soroban-testnet.stellar.org`). |
| `passphrase` | `string` | Hard-coded `Test SDF Network ; September 2015`. |

> **Category/limit note.** `categories`, `limits`, and `threshold` are fixed constants baked into the backend's `CHAIN` object — the demo always initializes new pools with the same 2-of-3, Equipment/Venue/Refreshments policy. The `/rules` endpoint's richer, AI-derived policy is a *display/onboarding* artifact; the on-chain `initialize` in `/pool/create` uses these fixed constants.

### 4.5 `POST /faucet`

Mints test USDC to a Stellar address by invoking `mint` on the USDC SAC, signed by the issuer identity from the local CLI keystore. **Precondition:** the target address must already hold a USDC trustline (classic-asset recipients cannot receive USDC without one — the browser establishes trustlines client-side during onboarding).

**Request body**

```json
{ "address": "G...", "amount": "100000000000" }
```

| Field | Type | Rules |
| --- | --- | --- |
| `address` | `string` | Required. Must start with `G` (classic account). Else **400** `Invalid Stellar address`. |
| `amount` | `string` | Optional. Default `"100000000000"` = **10,000 USDC** (7 decimals). Passed straight to the CLI. |

**Response `200`**: `{ "ok": true }`

**Errors**

| Status | Body | Cause |
| --- | --- | --- |
| `400` | `Invalid Stellar address` | `address` doesn't start with `G`. |
| `500` | CLI stderr/message | Mint failed (e.g. no trustline, bad SAC id). |

**CLI invoked**

```
stellar contract invoke --id <USDC_SAC_ID>
  --source kolektibo-usdc-issuer --network testnet
  -- mint --to <address> --amount <amount>
```

The frontend helper (`lib/backend.ts → faucet(address)`) always sends just `{ address }`, so the 10,000 USDC default applies.

### 4.6 `POST /pool/create`

Deploys a **fresh** treasury contract for a group and initializes it with the browser's officer public keys as the on-chain officers. This is the "create-your-own-pool" onboarding path — each group gets its own contract instance. Signed by the deployer identity from the local CLI keystore.

**Request body**

```json
{ "officers": ["G...ramon", "G...nena", "G...jun"], "threshold": 2 }
```

| Field | Type | Rules |
| --- | --- | --- |
| `officers` | `string[]` | Required. **At least 2** entries, each must start with `G`. Else **400** `Provide at least 2 officer public keys`. |
| `threshold` | `number` | Optional, default `2`. Number of approvals required to release a spend. |

**Response `200`**

```json
{ "ok": true, "contractId": "C..." }
```

**Errors**

| Status | Body | Cause |
| --- | --- | --- |
| `400` | `Provide at least 2 officer public keys` | `< 2` officers, or any officer not starting with `G`. |
| `500` | CLI stderr/message | Deploy or initialize failed. |

**Two CLI calls (deploy, then initialize)**

```
# 1) deploy the wasm → prints the new contract id
stellar contract deploy --wasm <TREASURY_WASM_PATH>
  --source kolektibo-deployer --network testnet

# 2) initialize it with the group's policy
stellar contract invoke --id <contractId>
  --source kolektibo-deployer --network testnet
  -- initialize
     --token      <USDC_SAC_ID>
     --officers   ["G...","G...","G..."]     # JSON.stringify(officers)
     --threshold  2
     --categories ["Equipment","Venue","Refreshments"]   # CHAIN.categories
     --limits     ["50000000000","30000000000","15000000000"]  # CHAIN.limits
```

The returned `contractId` is the stdout of the deploy call. The frontend (`lib/backend.ts → createPoolOnChain()`) reads `data.contractId` and persists it as the browser's active pool.

> `categories` and `limits` are always the backend's fixed constants — only `officers` and `threshold` come from the request. The `token` is always the configured USDC SAC.

## 5. OpenAI integration

Both AI endpoints use the OpenAI Chat Completions API via `openai@^6.45.0`, with the model resolved from `OPENAI_MODEL` (default **`gpt-4o-mini`** — cheap and capable, and the value in `.env.example`).

| Endpoint | Temperature | `response_format` | Validation |
| --- | --- | --- | --- |
| `/rules` | `0` (deterministic) | `{ type: 'json_object' }` | zod `PolicySchema` on the parsed output |
| `/ask` | `0.2` (slightly warmer) | none (free text) | none — answer returned as-is |

The client is instantiated once at boot and reused: `const client = apiKey ? new OpenAI({ apiKey }) : null`.

### System prompts (verbatim intent)

**`RULES_SYSTEM`** — instructs the model to emit *only* a JSON object with the exact policy shape, and encodes parsing conventions:

- Strip currency symbols and commas from amounts (`₱5,000 → 5000`).
- `"2 of 3 officers" → approval.threshold=2, approval.of=3`; **default `threshold=2, of=3`** if unspecified.
- `monthlyLimit = null` when a category has no cap; `dues = null` when none mentioned.
- Category names short and Title-Cased (e.g. `Equipment`, `Venue`, `Refreshments`).

**`ASK_SYSTEM`** — casts the model as *Kolektibo*, "the AI treasurer for a community pooled fund in the Philippines," with these guardrails:

- Warm, trustworthy, plain-language tone ("like a helpful barangay treasurer").
- **Use ONLY the on-chain state provided. Never invent numbers.**
- Amounts in Philippine pesos, formatted like `₱1,200`. Keep answers to **2–4 short sentences**.
- For "can we afford X" questions, compare X to balance and category limit and give a clear yes/no.
- If the state lacks the answer, say so honestly and suggest what's needed.

### Why zod on `/rules` but not `/ask`

`/rules` output is **structured data that drives on-chain initialization**, so a wrong shape must fail loudly (→ 422) rather than corrupt a policy. `/ask` output is **prose shown to a human**, so it's returned as-is; the grounding guarantee comes from the prompt discipline (state-only, no invented numbers), not a schema.

### OpenAI key requirement

- The AI endpoints are gated: if `OPENAI_API_KEY` is missing, `client === null` and both `/rules` and `/ask` short-circuit to **HTTP 500** `OPENAI_API_KEY not set on the AI service`.
- `/health` reports the key state (`hasKey`) and the boot log prints `key: set` / `key: MISSING`.
- The key is the service's **only genuine secret**. It lives in git-ignored `.env`, never in the frontend bundle (the browser talks to *this* service, which holds the key server-side).

## 6. The chain-ops model

The `/faucet` and `/pool/create` endpoints (and the CLI paths behind them) never touch a private key in code, env, or the transcript. Instead they **`execFile` the Stellar CLI**, which reads the signing keys from its own **local keystore** by *identity name*.

### The `stellar()` helper

```ts
async function stellar(args: string[]): Promise<string> {
  // Strip RPC/passphrase env vars so `--network testnet` fully defines the network
  // (a leaked SOROBAN_RPC_URL makes the CLI demand a matching passphrase).
  const env = { ...process.env }
  delete env.SOROBAN_RPC_URL
  delete env.STELLAR_RPC_URL
  delete env.STELLAR_NETWORK_PASSPHRASE
  delete env.SOROBAN_NETWORK_PASSPHRASE
  const { stdout } = await pExecFile(CHAIN.bin, args, {
    maxBuffer: 16 * 1024 * 1024,
    env,
  })
  return stdout.trim()
}
```

| Detail | Value / rationale |
| --- | --- |
| Executor | `promisify(execFile)` → `pExecFile` (no shell; argv array, so no shell-injection surface) |
| Binary | `CHAIN.bin` = `STELLAR_BIN` (default `C:/Program Files (x86)/Stellar CLI/stellar.exe`) |
| `maxBuffer` | `16 MiB` — deploy/invoke output can be large |
| Env stripping | Removes `SOROBAN_RPC_URL`, `STELLAR_RPC_URL`, `STELLAR_NETWORK_PASSPHRASE`, `SOROBAN_NETWORK_PASSPHRASE` |
| Return | Trimmed `stdout` (e.g. the deployed contract id) |

**Why strip those four env vars?** `.env` sets `SOROBAN_RPC_URL` so the client (`/config`) can use it — but if that variable leaks into the CLI child process, the CLI infers a custom network and then **demands a matching network passphrase**, breaking the call. Stripping them makes **`--network testnet` the single, authoritative network definition** for every CLI invocation.

### Error surfacing

CLI failures are normalized by `chainErr(e)`, which prefers `e.stderr`, then `e.message`, then `'chain op failed'`, and is returned as the HTTP 500 body so the client sees the real reason (missing trustline, bad SAC id, etc.).

### The `CHAIN` config object

```ts
const CHAIN = {
  bin:        process.env.STELLAR_BIN || 'C:/Program Files (x86)/Stellar CLI/stellar.exe',
  network:    process.env.STELLAR_NETWORK || 'testnet',
  usdcSac:    process.env.USDC_SAC_ID   || '',
  usdcIssuer: process.env.USDC_ISSUER   || '',
  issuer:     process.env.ISSUER_IDENTITY   || 'kolektibo-usdc-issuer',
  deployer:   process.env.DEPLOYER_IDENTITY || 'kolektibo-deployer',
  wasmPath:   process.env.TREASURY_WASM_PATH || '',
  rpcUrl:     process.env.SOROBAN_RPC_URL || 'https://soroban-testnet.stellar.org',
  passphrase: 'Test SDF Network ; September 2015',
  categories: ['Equipment', 'Venue', 'Refreshments'],
  limits:     ['50000000000', '30000000000', '15000000000'], // 5000/3000/1500 USDC * 1e7
}
```

### The no-secrets rationale

```
┌──────────────┐   HTTP    ┌─────────────────────┐  execFile   ┌───────────────┐
│  Browser     │ ────────▶ │ services/ai backend │ ──────────▶ │ Stellar CLI   │
│ (personas    │  no keys  │ (holds identity     │ identity    │ + local       │
│  sign client │  in body  │  *names*, not keys) │  by name    │ keystore      │
│  side)       │           │                     │             │ (real keys)   │
└──────────────┘           └─────────────────────┘             └───────────────┘
```

- The backend references signing identities **by name only** (`kolektibo-usdc-issuer`, `kolektibo-deployer`). The actual secret keys live in the Stellar CLI keystore on the machine and are **never materialized** into code, `.env`, the API surface, or logs.
- During development, an attempt to chain a `stellar keys show` (which would have printed a secret) was deliberately blocked by the harness and avoided — no private key ever entered the transcript.
- The browser's **officer personas** (Kap. Ramon, Aling Nena, Kuya Jun) are separate in-browser keypairs that sign **contribute / request_spend / approve / execute** client-side via `basicNodeSigner`. Their secrets never leave the device and never reach this backend.
- Division of labor: the backend only does the two operations that inherently need issuer/deployer authority — **minting test USDC** and **deploying + initializing new treasuries**. It has **zero authority to move money out of an existing pool**; that is gated entirely by the contract's on-chain 2-of-3 multisig.

## 7. Consolidated error reference

| Endpoint | Status | Body | Trigger |
| --- | --- | --- | --- |
| `/rules` | 400 | `Missing "text"` | Empty/absent `text` |
| `/rules` | 422 | zod/JSON message | Model output fails `PolicySchema` |
| `/rules` | 500 | `OPENAI_API_KEY not set on the AI service` | No OpenAI key |
| `/ask` | 400 | `Missing "question"` | Empty/absent `question` |
| `/ask` | 500 | `OPENAI_API_KEY not set on the AI service` | No OpenAI key |
| `/ask` | 500 | error message | OpenAI call failed |
| `/faucet` | 400 | `Invalid Stellar address` | `address` not starting with `G` |

| `/pool/create` | 400 | `Provide at least 2 officer public keys` | `< 2` officers or bad pubkey |

Errors are logged server-side with a tagged prefix (`[/rules]`, `[/ask]`, `[/faucet]`, `[/pool/create]`) before the response is sent.

## 8. End-to-end request flows

**Onboarding a new pool (browser → backend → chain):**

```
1. Browser generates/loads 3 officer personas (localStorage keypairs)
2. Browser funds + trustlines personas via Friendbot / change_trust (client-side)
3. POST /pool/create { officers:[3 pubkeys], threshold:2 }
      → stellar contract deploy   (deployer identity)  → contractId
      → stellar contract invoke initialize(...)         (deployer identity)
   ← { ok:true, contractId }
4. POST /faucet { address: <treasury or persona> }      (issuer identity mints USDC)
5. Browser signs contribute / request_spend / approve / execute directly (personas)
6. GET /config  → RPC, passphrase, USDC SAC, category policy for client reads
7. POST /ask { question, state: <live on-chain state> } → grounded plain-language answer
```

The moat (collecting 2-of-3 approvals and releasing USDC) happens **on-chain via the browser-signed persona transactions in step 5** — the backend is not in that trust path.

## 10. Design notes & rationale

- **Stateless by design.** No database. The blockchain is the database for all trust-critical state; the browser's `localStorage` holds client wallet state. An off-chain DB would only ever be warranted later for *non-trust* metadata (human names, a pool directory, cross-device membership, notifications, event indexing) and would hold **zero authority over money**.
- **OpenAI, not Claude.** The AI provider is OpenAI by explicit team choice; default model `gpt-4o-mini` for cost + capability.
- **Two hats, one process.** Colocating the AI treasurer and the thin chain-ops backend keeps the demo footprint to a single `tsx` process on port `8787`, while the security boundary (browser signs its own money moves; backend only mints test USDC and deploys new pools) stays intact.
- **`--network testnet` is authoritative.** The env-stripping in `stellar()` is the fix for a real bug: a leaked `SOROBAN_RPC_URL` made the CLI demand a passphrase. Every chain op now depends only on `--network testnet`.

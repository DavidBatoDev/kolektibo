# System Architecture
_Created: 2026-07-11 01:46 MPST · Kolektibo · Stellar Testnet_

> **Kolektibo** — an AI-governed group treasury on Stellar. The AI does the labor of a
> human treasurer (tracks contributions and balances, drafts disbursements, answers
> questions in plain language). The money lives in a Soroban smart contract that only ever
> releases funds when the group's own on-chain policy is satisfied — the right category,
> within its limit, with the required officer approvals collected. **No single officer, and
> not the AI, can drain the fund.** Settlement is in USDC on Stellar.

This document is the authoritative technical reference for the build team and hackathon
judges. Every name, signature, version, and contract ID below is taken directly from the
repository, not paraphrased.

---

## 1. The one-paragraph mental model

There are four moving parts and exactly one of them holds authority over money:

- **`apps/web`** — a mobile-first React PWA. It reads chain state, renders the treasury, and
  signs write transactions **client-side** using in-browser officer keypairs. It never asks a
  server to move money.
- **`contracts/treasury`** — the Soroban (Rust) contract. **This is the trust core and the
  database.** It custodies the USDC and enforces the M-of-N approval + category-limit policy.
- **`services/ai`** — a stateless Node/Express backend. It does two unprivileged jobs: (a) LLM
  work (natural-language rules → policy JSON, and grounded Q&A), and (b) thin chain-ops
  (deploy/initialize a pool, faucet test USDC) using identities held in the **local Stellar CLI
  keystore**. It holds no application state and never touches user funds.
- **Stellar Testnet** — the settlement layer. USDC exists as a Stellar Asset Contract (SAC);
  accounts are funded by Friendbot; contract calls go through the Soroban RPC.

The security property that makes this trustworthy is not in the AI or the web app — it is that
**every money movement is a `require_auth`'d contract call whose release path
(`execute`) reverts unless `approvals >= threshold`.** The AI can propose, summarize, and
explain, but it can only ever *call* a contract that refuses to over-spend.

## 3. Component responsibilities

### 3.1 `contracts/treasury` — the Soroban trust core (Rust)

One deployed instance **is one pool**. Source: `contracts/treasury/src/lib.rs`
(`#![no_std]`, `soroban_sdk` 26). The contract type is `TreasuryContract`.

**Responsibilities**
- Custody the group's USDC (held as the contract's own balance in the USDC SAC).
- Enforce, on-chain and unconditionally, the group's policy:
  - only registered officers may propose or approve a spend;
  - a spend cannot exceed its category limit (when the limit is non-zero);
  - funds are released only when `approvals >= threshold`.
- Record the audit trail: every contribution, every spend request, every approval, and the
  executed flag — all queryable, all the source of truth for the app and the AI.

**Storage (`DataKey` enum) — this is the database:**

| Key | Kind | Holds |
|---|---|---|
| `Token` | instance | USDC SAC address |
| `Officers` | instance | `Vec<Address>` of officer public keys |
| `Threshold` | instance | `u32` — approvals required to release |
| `NextSpendId` | instance | `u32` — monotonic spend counter (starts at 1) |
| `Members` | instance | `Vec<Address>` — everyone who has contributed |
| `Categories` | instance | `Vec<Symbol>` — spend category names |
| `CategoryLimit(Symbol)` | instance | `i128` — per-spend cap (0 = no limit) |
| `Contribution(Address)` | persistent | `i128` — running total per member |
| `Spend(u32)` | persistent | `SpendRequest` — a proposed/executed disbursement |

**Structs**

```rust
pub struct CategoryInfo { pub name: Symbol, pub limit: i128 }   // 0 = no limit

pub struct SpendRequest {
    pub id: u32,
    pub proposer: Address,
    pub category: Symbol,
    pub amount: i128,
    pub recipient: Address,
    pub memo: String,
    pub approvals: Vec<Address>,
    pub executed: bool,
}
```

**Full interface (exact signatures)**

| Function | Auth / who | Behavior |
|---|---|---|
| `initialize(token: Address, officers: Vec<Address>, threshold: u32, categories: Vec<Symbol>, limits: Vec<i128>)` | deployer, one-time | Reverts `AlreadyInitialized` if set; reverts `InvalidInit` if `threshold == 0 \|\| threshold > officers.len() \|\| categories.len() != limits.len()`. Seeds storage, `NextSpendId = 1`, empty members, extends TTL 100_000. |
| `contribute(from: Address, amount: i128)` | `from.require_auth()` | Reverts `NonPositiveAmount` if `amount <= 0`. Transfers `amount` of the token from `from` into the contract, adds to `Contribution(from)`, adds `from` to `Members` if new. Emits `contrib`. |
| `request_spend(proposer: Address, category: Symbol, amount: i128, recipient: Address, memo: String) -> u32` | `proposer.require_auth()` + must be officer | Reverts `NotOfficer`, `NonPositiveAmount`, or `OverCategoryLimit` (only when the stored limit `> 0`). Auto-records the proposer's own approval, stores the `SpendRequest`, returns its `id`, increments `NextSpendId`. Emits `spend_req`. |
| `approve(officer: Address, spend_id: u32)` | `officer.require_auth()` + must be officer | Reverts `SpendNotFound`, `AlreadyExecuted`, or `AlreadyApproved`. Appends the officer to `approvals`. Emits `approve`. |
| `execute(spend_id: u32)` | **permissionless** | Reverts `SpendNotFound`, `AlreadyExecuted`, `NotEnoughApprovals` (if `approvals.len() < threshold`), or `InsufficientBalance`. Transfers `amount` USDC to `recipient`, sets `executed = true`. Emits `execute`. |
| `get_balance() -> i128` | view | Contract's USDC balance. |
| `get_spend(id: u32) -> Option<SpendRequest>` | view | One spend. |
| `get_spends() -> Vec<SpendRequest>` | view | All spends 1..NextSpendId. |
| `get_categories() -> Vec<CategoryInfo>` | view | Categories + caps (feeds budgets UI + AI). |
| `get_officers() -> Vec<Address>` | view | Officers. |
| `get_threshold() -> u32` | view | Approvals required. |
| `get_members() -> Vec<Address>` | view | Contributors. |
| `get_contribution(who: Address) -> i128` | view | One member's total. |
| `get_next_spend_id() -> u32` | view | Next id. |

**Error codes (`#[contracterror] repr(u32)`):** `1 AlreadyInitialized`, `2 NotOfficer`,
`3 OverCategoryLimit`, `4 SpendNotFound`, `5 AlreadyExecuted`, `6 NotEnoughApprovals`,
`7 AlreadyApproved`, `8 InvalidInit`, `9 InsufficientBalance`, `10 NonPositiveAmount`.

**Events** (`env.events().publish`): `contrib (from, amount)`, `spend_req (id)`,
`approve (spend_id, officer)`, `execute (spend_id, amount)`.

**Why approvals cannot be faked:** each `approve` (and `request_spend`) call is
`require_auth`'d against the officer's own address, so the contract cryptographically knows the
approval came from that officer — there is no impersonation and no server in the trust path.

### 3.2 `services/ai` — stateless AI + chain-ops backend (Node + Express 5)

Source: `services/ai/src/index.ts`. Listens on `PORT` (default **8787**). It exists so the
`OPENAI_API_KEY` and the deploy/faucet identities never ship to the browser. **It holds no
application state and no user funds.**

| Endpoint | Purpose | Notes |
|---|---|---|
| `GET /health` | Liveness | `{ ok, model, hasKey }` |
| `POST /rules { text }` | NL rules → policy | OpenAI `gpt-4o-mini`, `temperature: 0`, `response_format: json_object`; result validated by a zod `PolicySchema` → `{ currency, dues, categories[], approval{threshold,of}, summary }`. Returns `{ policy }` (422 on parse failure). |
| `POST /ask { question, state }` | Grounded Q&A | OpenAI `temperature: 0.2`; system prompt forces answers **only** from the passed-in on-chain `state`, warm plain language, pesos, 2–4 sentences. Returns `{ answer }`. |
| `GET /config` | Public chain config | `{ network, usdcSac, usdcIssuer, rpcUrl, passphrase, friendbotUrl, categories, limits, threshold, configured }` — no secrets. |
| `POST /faucet { address, amount? }` | Mint test USDC | `stellar contract invoke … mint --to <address>` via the **issuer** CLI identity; address must already trust USDC. Default amount `100000000000` (10,000 USDC). |
| `POST /pool/create { officers[], threshold }` | Deploy a fresh pool | `stellar contract deploy` then `initialize` via the **deployer** CLI identity; returns `{ contractId }`. Requires ≥ 2 officer `G…` keys. |

Chain ops go through a `stellar(args)` helper that `execFile`s the Stellar CLI binary
(`STELLAR_BIN`, default `C:/Program Files (x86)/Stellar CLI/stellar.exe`). Before invoking it
**strips `SOROBAN_RPC_URL` / `STELLAR_RPC_URL` / `*_NETWORK_PASSPHRASE` from the child env** so
that `--network testnet` is the single authoritative network definition (a leaked RPC URL made
the CLI demand a matching passphrase).

### 3.3 `apps/web` — mobile-first PWA (React 19 + Vite 8)

Source under `apps/web/src`. Capacitor-ready static build; `vite-plugin-pwa` for installability.
It is the only place money movements are initiated, and it signs them locally.

**Key modules**

- `lib/wallet.ts` — the three officer personas **Kap. Ramon**, **Aling Nena**, **Kuya Jun** as
  in-browser `Keypair.random()` keypairs persisted in `localStorage` under
  `kolektibo.personas.v1`. Secrets are generated on-device and **never leave it**. Exposes
  `getPersonas`, `personaByName`, `keypairFor`, `resetPersonas`.
- `lib/contract.ts` — `TREASURY_ID` from `VITE_TREASURY_CONTRACT_ID` (else the canonical
  bindings id); `SCALE = 10_000_000`; `rawToUsd`/`usdToRaw`; `activePoolId()` returns the
  browser-created pool id (`localStorage['kolektibo.contract']`) or the canonical one;
  `readClient()`, `fetchLiveTreasury()`, `contractExplorerUrl()`.
- `lib/livepool.ts` — the live money layer. `client(contractId, signerKp?)` builds a generated
  `Client`; when a keypair is passed it wires `basicNodeSigner(kp, passphrase)` into
  `signTransaction`/`signAuthEntry`. Contains `ensureFunded` (Friendbot), `ensureTrustline`
  (`changeTrust` to USDC), the one-time `createPool(onProgress)`, the chain-signed writes
  `contribute` / `requestSpend` / `approve` / `execute`, and `readPool()` which assembles the
  `Pool` view model (names from personas, amounts via `rawToUsd`).
- `lib/stellar.ts` — `NETWORK` (passphrase `Networks.TESTNET`, Soroban RPC, Horizon,
  Friendbot), `Horizon.Server`, `rpc.Server`, `fundWithFriendbot`, `getAccountSummary`,
  explorer URL helpers.
- `lib/ai.ts` / `lib/backend.ts` — thin fetch clients for the backend (`parseRules`, `askAI`;
  `getConfig`, `faucet`, `createPoolOnChain`). `AI_URL` defaults to `http://localhost:8787`.
- `hooks/usePool.ts` — TanStack Query: `usePool` (queryKey `['pool']`, `staleTime 5_000`,
  `retry 1`), `useHasPool`, `usePoolBalance`, `useCreatePool` (with progress), and
  `usePoolActions` (the four write mutations, each invalidating `['pool']` on success).
- `contract/treasury/src/index.ts` — the generated TypeScript bindings (`Client`, `networks`,
  `CategoryInfo`, `SpendRequest`, `Errors`). `networks.testnet.contractId` is the canonical
  treasury.

**Components / routes:** `AppShell` (mobile shell + bottom nav Pool / Contribute / Spend /
Rules), `CreatePool` (deploy onboarding), `LiveOnChain` (live on-chain card, 15s refetch),
`ui` (Card/Button/peso helpers); routes `Home` (balance hero, LiveOnChain, Ask-Treasurer AI
box, category budgets, activity), `Contribute` (persona picker + contribute), `Spend` (request
form + per-officer approval chips + a Release button gated on threshold), `Setup` (AI rules
parse + officers list + start over).

## 4. Data-flow walkthroughs

### 4.1 Read path — "how much do we have, and where did it go?"

Reads never sign and never hit the AI backend for chain state; they simulate view calls
directly against the Soroban RPC.

```
Home route / LiveOnChain
      │  useQuery(['pool'])  →  livepool.readPool()          (hooks/usePool.ts)
      ▼
readPool()  →  client(id)                                     (no signer)
      │  Promise.all([ get_balance, get_threshold, get_officers,
      │                get_categories, get_members, get_spends ])
      ▼
Soroban RPC  simulateTransaction  for each view               (contracts/treasury)
      │  + get_contribution(who) for each member
      ▼
Pool view model  { name, currency, policy, officers, members, spends }
      │  amounts via rawToUsd (÷ 1e7), addresses → persona names
      ▼
React renders balance hero, budgets, activity, approval chips
```

The **Ask-the-Treasurer** box is a second, independent read: the client passes the freshly
fetched on-chain `state` to `askAI(question, state)` → `POST /ask`. The LLM answers **only**
from that state (system prompt forbids inventing numbers), so the AI is a narrator over
on-chain truth, never a source of it.

### 4.2 Write path — the winning demo loop, on-chain

Every write is signed **in the browser** with the acting persona's keypair, assembled by the
generated bindings, and submitted to the Soroban RPC. The contract enforces policy; the server
is not in this path.

```
Spend route → usePoolActions().requestSpend / approveSpend / executeSpend
      │
      ▼  livepool.requestSpend("Kap. Ramon", "Equipment", amount, memo)
client(id, kp)  ── basicNodeSigner(kp, passphrase) ──►  request_spend AssembledTransaction
      │  .signAndSend()                                       (proposer.require_auth)
      ▼
Soroban contract:  require_auth ✔  officer ✔  amount ≤ category limit ✔
      │  stores SpendRequest, auto-approval by proposer, emits spend_req
      ▼
approve("Aling Nena", id)  →  second officer signs  →  approvals = 2   (of 3)
      ▼
execute(id)   ── PERMISSIONLESS ──►  approvals(2) ≥ threshold(2) ✔
      │  transfers USDC to recipient, executed = true, emits execute
      ▼  returns tx hash  →  TanStack Query invalidates ['pool']  →  UI re-reads
```

The gate is in the contract, not the UI: the Release button is *shown* only at threshold, but
even if `execute` were called early it would revert with `NotEnoughApprovals (6)`. Duplicate
approvals revert `AlreadyApproved (7)`; a non-officer proposal reverts `NotOfficer (2)`; an
over-limit request reverts `OverCategoryLimit (3)`.

### 4.3 Onboarding path — `createPool(onProgress)` (create-your-own pool)

```
getConfig() ─► for each persona: ensureFunded (Friendbot) → ensureTrustline (changeTrust USDC)
            ─► fund + trustline the payee account
            ─► createPoolOnChain(officers, threshold)  →  POST /pool/create
                    backend: stellar contract deploy  +  initialize (deployer identity)
            ─► setPoolId(contractId)   (localStorage 'kolektibo.contract')
            ─► for each persona: faucet(pk)  →  POST /faucet  (issuer mints test USDC)
            ─► seed real contributions: Kap. Ramon ₱600, Aling Nena ₱400, Kuya Jun ₱600
```

This mirrors the verified `smoke-write.mts` E2E: create pool via backend, fund + trustlines,
faucet, contribute (client-signed), request_spend, approve to 2-of-3, execute.

## 6. "Our database is the blockchain"

Kolektibo ships **no application database**. Every piece of state that carries authority over
money lives on-chain; the browser holds only client identity/UI state; the backend holds
nothing. This is a deliberate trust decision — you cannot silently rewrite a balance, an
approval, or a spend record, because the only writer is a `require_auth`'d contract call.

| State | Where it lives | Why there | Authority over money |
|---|---|---|---|
| USDC balance, officers, threshold, categories + limits, members, per-member contributions, spend requests + approvals + executed flag | **Soroban contract storage** (instance + persistent) | It is trust-critical and must be tamper-evident, auditable, and enforced without a trusted server | **Full** — this is the source of truth |
| Officer persona keypairs (secrets) | Browser `localStorage` `kolektibo.personas.v1` | Signing must happen on the user's device; secrets must never transit a server or the bundle | None directly — they are the *means* to authorize, but the contract still enforces policy |
| Active pool id | Browser `localStorage` `kolektibo.contract` | Which pool this browser created; purely a pointer | None |
| Payee account keypair | Browser `localStorage` `kolektibo.payee.v1` | Demo recipient for spends | None |
| Query cache (pool view model) | In-memory TanStack Query (`['pool']`) | Caching/refetch/loading UX only; re-derived from chain | None |
| OpenAI key, CLI identities | Backend env / **local CLI keystore** | Must stay off the client | None over the treasury; can deploy/faucet only |

**Why no server DB:** the blockchain already provides consensus, durability, an audit log, and
authorization for exactly the data that matters. A second copy in Postgres would be a weaker,
forgeable duplicate of the truth. The backend is intentionally **stateless**: it reads config
from env, calls OpenAI, and shells out to the Stellar CLI — nothing persists between requests.

**What an off-chain DB would add later (with zero money authority):** human-readable names,
a pool directory / discovery, cross-device membership, notifications, and event indexing for
fast history. All of it is *non-trust metadata* — it would annotate on-chain facts but never
gate a disbursement. The contract would remain the sole arbiter of funds.

## 7. Key-handling & security model

The system is designed so that **no private key and no secret ever appears in the shipped
bundle, the environment sent to the browser, or a transcript.** There are two distinct key
domains, and they never overlap.

### 7.1 Client domain — officer personas sign in the browser

- The three officer personas are generated with `Keypair.random()` in `lib/wallet.ts` and
  stored in `localStorage`. **Secrets are created on-device and never leave it.**
- Writes are signed **client-side** via `basicNodeSigner(kp, NETWORK.passphrase)`, which
  supplies `signTransaction` and `signAuthEntry` to the generated `Client`. The signed
  transaction goes straight to the Soroban RPC — no server ever sees a secret or a signed money
  transaction it didn't need to.
- Because each `contribute` / `request_spend` / `approve` is `require_auth`'d against the
  signer's own address, the contract verifies authorship cryptographically. `execute` is
  permissionless by design — anyone can push the button *once officers have already signed*, and
  it still reverts below threshold.

### 7.2 Backend domain — CLI keystore, no materialized secrets

- The backend's privileged operations (deploy/initialize a pool, mint test USDC) use the
  identities `kolektibo-deployer` and `kolektibo-usdc-issuer`, which live in the **local Stellar
  CLI keystore**. The backend invokes `stellar.exe` via `execFile` and passes only identity
  *names* — it never reads, prints, or embeds the secret keys.
- A chained `stellar keys show` (which would have materialized a secret) was **correctly blocked
  by the tooling and deliberately avoided.** No secret key is ever written to code, `.env`, the
  transcript, or the browser bundle.
- The only backend secret is `OPENAI_API_KEY`, which stays server-side; `GET /config` returns
  strictly public values.
- `GET /faucet` and `POST /pool/create` are unprivileged with respect to the treasury: they can
  create/fund pools and mint *test* USDC, but they cannot approve or release a spend — that path
  exists only inside the contract.

### 7.3 What this buys the group

The AI can be wrong, biased, or compromised, and a single officer can be careless or malicious —
and the fund is still safe, because releasing money requires (a) the correct category within its
limit and (b) `threshold` real officer signatures, enforced by code that no participant can edit
at runtime. That guarantee has been demonstrated end-to-end on testnet.

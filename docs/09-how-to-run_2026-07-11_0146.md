# How to Run & Demo
_Created: 2026-07-11 01:46 MPST В· Kolektibo В· Stellar Testnet_

Everything you need to stand up Kolektibo on a fresh machine, prove the smart
contract on-chain, run the two dev servers, and walk a judge through the winning
demo loop вҖ” **create pool вҶ’ contribute вҶ’ request вҶ’ approve вҶ’ execute вҶ’ ask the AI.**

Kolektibo is an **AI-governed group treasury on Stellar**: an AI treasurer does the
labour, but the money lives in a **Soroban smart contract** that only releases funds
when the group's on-chain policy is satisfied (right category, within limit, enough
officer approvals). The moat is the *approve вҶ’ execute* gate, not the AI.

> **The demo is currently single-browser.** All three officer personas
> (Kap. Ramon, Aling Nena, Kuya Jun) are in-browser testnet keypairs kept in
> `localStorage`, and each signs its own approval locally. This is deliberate for a
> one-laptop demo; multi-device (each officer on their own phone) is a documented
> stretch, not built. Present from a single browser tab.

## 3. Environment setup

Two env files. The **AI service** file you create by hand; the **web** file is written
for you by the deploy script.

### 3a. `services/ai/.env` вҖ” AI + chain-ops backend

Copy the example and add your key:

```bash
cp services/ai/.env.example services/ai/.env
```

`.env.example` ships with:

```dotenv
OPENAI_API_KEY=sk-your-key-here
# Any chat model you have access to. Cheap + capable default:
OPENAI_MODEL=gpt-4o-mini
PORT=8787
```

| Var | Default | Required for | Notes |
|---|---|---|---|
| `OPENAI_API_KEY` | вҖ” | `POST /rules`, `POST /ask` | Without it, `/health` returns `hasKey:false` and those routes 500. |
| `OPENAI_MODEL` | `gpt-4o-mini` | AI calls | Any chat model you can access. |
| `PORT` | `8787` | server | The web app's `VITE_AI_URL` must match. |

**Chain-ops env (optional but needed for in-app "Create pool").** `services/ai/src/index.ts`
also reads the following to power `POST /faucet` and `POST /pool/create` (the browser's
create-your-own-pool onboarding). They have code defaults; set them explicitly if the
defaults don't match your machine. The endpoint is only advertised as `configured`
when both `USDC_SAC_ID` and `TREASURY_WASM_PATH` are present.

| Var | Code default | Purpose |
|---|---|---|
| `STELLAR_BIN` | `C:/Program Files (x86)/Stellar CLI/stellar.exe` | Path to the Stellar CLI the backend shells out to |
| `STELLAR_NETWORK` | `testnet` | Network passed as `--network` |
| `USDC_SAC_ID` | `''` | Test USDC Stellar Asset Contract id (from deploy) |
| `USDC_ISSUER` | `''` | Classic USDC issuer public key (returned by `/config`) |
| `ISSUER_IDENTITY` | `kolektibo-usdc-issuer` | Local CLI keystore identity that mints faucet USDC |
| `DEPLOYER_IDENTITY` | `kolektibo-deployer` | Local CLI keystore identity that deploys pools |
| `TREASURY_WASM_PATH` | `''` | Path to `treasury.wasm` for `/pool/create` |
| `SOROBAN_RPC_URL` | `https://soroban-testnet.stellar.org` | Reported in `/config` |

> **No secrets in code, env, or bundle.** The issuer/deployer *identities* live in the
> local Stellar CLI keystore; the backend only references them by name. The browser
> officer personas sign client-side. Nothing here materialises a private key.

After `pnpm contract:deploy`, populate `USDC_SAC_ID` and `TREASURY_WASM_PATH` in
`services/ai/.env` if you want the in-app "Deploy on Stellar testnet" button to work:

```dotenv
USDC_SAC_ID=CDTCIZLKSZNDFDSZRQUFIHQ5P5L2OOI5DDOMSY5NH6NQQTGSOE5LK7QR
TREASURY_WASM_PATH=contracts/target/wasm32v1-none/release/treasury.wasm
```

### 3b. `apps/web/.env.local` вҖ” web app config

**Written automatically by `scripts/deploy.sh` (step 6/6).** You normally never edit it.
Its shape (from `apps/web/.env.example`) is:

```dotenv
# All optional вҖ” sensible testnet defaults are built in.
VITE_AI_URL=http://localhost:8787
VITE_HORIZON_URL=https://horizon-testnet.stellar.org
VITE_SOROBAN_RPC_URL=https://soroban-testnet.stellar.org

# Filled in by scripts/deploy.sh once the contract is on testnet:
VITE_TREASURY_CONTRACT_ID=
VITE_USDC_SAC_ID=
```

| Var | Consumed by | Notes |
|---|---|---|
| `VITE_AI_URL` | `lib/ai.ts`, `lib/backend.ts` | Must equal the AI service origin (`http://localhost:8787`). |
| `VITE_HORIZON_URL` | `lib/stellar.ts` | Horizon (classic ops: fund, trustlines). |
| `VITE_SOROBAN_RPC_URL` | `lib/stellar.ts` | Soroban RPC (contract calls). |
| `VITE_TREASURY_CONTRACT_ID` | `lib/contract.ts` `TREASURY_ID` | Canonical demo pool; falls back to the value baked into the generated bindings if unset. |
| `VITE_USDC_SAC_ID` | web | Test USDC SAC id. |

> The **main pool view** (`hooks/usePool.ts` вҶ’ `lib/livepool.ts`) reads the
> *browser-created* pool id from `localStorage['kolektibo.contract']`. The
> **LiveOnChain card** (`lib/contract.ts` `activePoolId()`) falls back to
> `VITE_TREASURY_CONTRACT_ID` (the canonical pool) when no browser pool exists вҖ” so
> that card shows real chain state even before you click "Create pool".


## 5. Contract: build & test

The Rust workspace lives in `contracts/` (`treasury` crate). It is built with the
Stellar CLI / cargo, **not** pnpm.

### Build (wasm)

```bash
pnpm contract:build        # = cd contracts/treasury && stellar contract build
```

Output: `contracts/target/wasm32v1-none/release/treasury.wasm`. The release profile in
`contracts/Cargo.toml` is size-optimized: `opt-level = "z"`, `lto = true`,
`panic = "abort"`, `strip = "symbols"`, `codegen-units = 1`.

### Test

```bash
cd contracts
cargo test          # 4/4 pass
```

Four tests in `contracts/treasury/src/test.rs` prove the trust core:

| Test | Proves | Asserts |
|---|---|---|
| `full_happy_path` | contribute вҶ’ request вҶ’ approve to threshold вҶ’ execute releases USDC | balances move correctly |
| `cannot_execute_without_threshold` | one officer can't drain the fund | `Error #6 NotEnoughApprovals` |
| `rejects_over_category_limit` | category caps are enforced at request time | `Error #3 OverCategoryLimit` |
| `non_officer_cannot_propose` | only officers can propose spends | `Error #2 NotOfficer` |

> **If `cargo test` fails to build** with an `ed25519-dalek` error: the newly-released
> `ed25519-dalek 3.0.0` breaks `soroban-env-host` testutils. Pin it (tests only; the
> deployable wasm is unaffected):
> ```bash
> cargo update -p ed25519-dalek --precise 2.2.0
> ```

### Contract surface (for reference)

One contract instance = one pool. Storage keys (`DataKey`): `Token`, `Officers`,
`Threshold`, `NextSpendId`, `Members`, `Categories`, `CategoryLimit(Symbol)`,
`Contribution(Address)`, `Spend(u32)`.

```rust
initialize(token, officers: Vec<Address>, threshold: u32, categories: Vec<Symbol>, limits: Vec<i128>)
contribute(from: Address, amount: i128)                       // from.require_auth()
request_spend(proposer, category, amount, recipient, memo) -> u32  // must be officer; checks limit; auto-approves proposer
approve(officer: Address, spend_id: u32)                      // must be officer; no double-approve
execute(spend_id: u32)                                        // PERMISSIONLESS; reverts unless approvals >= threshold
// views: get_balance, get_spend, get_spends, get_categories, get_officers,
//        get_threshold, get_members, get_contribution, get_next_spend_id
```

Error codes (`contracterror`, repr u32): `1 AlreadyInitialized`, `2 NotOfficer`,
`3 OverCategoryLimit`, `4 SpendNotFound`, `5 AlreadyExecuted`, `6 NotEnoughApprovals`,
`7 AlreadyApproved`, `8 InvalidInit`, `9 InsufficientBalance`, `10 NonPositiveAmount`.

## 6. Deploy to testnet

```bash
pnpm contract:deploy        # = bash scripts/deploy.sh
```

`scripts/deploy.sh` runs 6 steps on **Stellar Testnet**:

```
в–¶ 1/6  Ensure funded identities (Friendbot):
        kolektibo-deployer, -officer2, -officer3, -usdc-issuer
в–¶ 2/6  stellar contract build вҶ’ treasury.wasm
в–¶ 3/6  Ensure test USDC exists as a Stellar Asset Contract (deterministic id; deploy once, reuse)
в–¶ 4/6  Deploy the treasury wasm  вҶ’ TREASURY_ID
в–¶ 5/6  initialize:
          token      = USDC SAC
          officers   = [deployer, officer2, officer3]
          threshold  = 2                    (2-of-3)
          categories = Equipment / Venue / Refreshments
          limits     = 50000000000 / 30000000000 / 15000000000   (5000 / 3000 / 1500 USDC Г— 1e7)
        + mint 20,000 USDC (200000000000 raw) into the treasury for the demo
в–¶ 6/6  Write apps/web/.env.local (VITE_TREASURY_CONTRACT_ID, VITE_USDC_SAC_ID, RPC/Horizon URLs)
```

On success it prints the treasury id, the USDC SAC id, and an explorer link. Re-running
deploys a **fresh** instance with new ids and rewrites `.env.local`.

### Units convention (important)

Test USDC is a SAC with **7 decimals**. The contract stores raw `i128`. Convention
across app + deploy: `raw = human Г— 10_000_000` (`SCALE = 1e7`). For the demo,
**1 USDC вүҲ вӮұ1**. So вӮұ200 dues вҶ’ `2000000000`, the 20,000 seed вҶ’ `200000000000`, the
Equipment вӮұ5,000 cap вҶ’ `50000000000`. The frontend formats raw вҶ’ вӮұ and multiplies by
`SCALE` on every write (`lib/contract.ts`: `usdToRaw` / `rawToUsd`).

### The winning loop, end to end

```
Create pool в”Җв–¶ Contribute в”Җв–¶ Request spend в”Җв–¶ Approve (Г—2 of 3) в”Җв–¶ Execute в”Җв–¶ Ask the AI
 (deploy)      (USDC in)     (officer, in-limit)  в”Җв”Җ the moat в”Җв”Җ    (USDC out)  (grounded answer)
```

#### Step 1 вҖ” Create the pool (Pool tab)

If no pool exists yet, the **Pool** tab shows the **"Create your group treasury"** card
(`components/CreatePool.tsx`). Click **"Deploy on Stellar testnet."** With progress
messages, the app (`lib/livepool.ts` `createPool`):

1. funds each persona + a payee account via Friendbot,
2. adds a USDC **trustline** to each (`change_trust`),
3. calls `POST /pool/create` вҶ’ backend deploys a fresh treasury (2-of-3) and `initialize`s it,
4. faucets test USDC to each persona (`POST /faucet`),
5. seeds three real on-chain contributions: **Kap. Ramon вӮұ600, Aling Nena вӮұ400, Kuya Jun вӮұ600**.

Takes ~1 minute. Requires the AI service to be running **and** chain-config
(`USDC_SAC_ID` + `TREASURY_WASM_PATH`) set вҖ” otherwise it throws
`Backend chain config missing`.

> **Shortcut for a time-boxed pitch:** you can skip in-app deployment and point the app
> at the already-seeded canonical treasury via `VITE_TREASURY_CONTRACT_ID`; the
> **LiveOnChain** card renders its real state immediately. But the *browser-created*
> pool is what drives Contribute/Spend, so for the full loop, create the pool.

After creation the **Pool** dashboard (`routes/Home.tsx`) shows: the balance hero
("вӮұвҖҰ В· settles in USDC on Stellar"), the **LiveOnChain** card (real chain state,
refetched every ~15s), the **Ask your AI treasurer** box, monthly category budgets with
progress bars, and recent activity.

#### Step 2 вҖ” Contribute (Contribute tab)

`routes/Contribute.tsx`. Pick a persona in **"Contributing as,"** choose an amount
(quick chips вӮұ100 / вӮұ200 / вӮұ500, dues default вӮұ200), tap **"Contribute вӮұN."** The app
signs `contribute(from, amount)` client-side and submits to Stellar. On success:
**"Contribution confirmed on-chain вң“"** and the **Members & contributions** list updates
from chain state. Point out: real USDC moved into the treasury contract.

#### Step 3 вҖ” Request a spend (Spend tab)

`routes/Spend.tsx`. Tap **"Request a spend."** Fill:
- **Proposed by (officer)** вҖ” e.g. Kap. Ramon,
- **Category** вҖ” Equipment (вүӨ вӮұ5,000) / Venue (вүӨ вӮұ3,000) / Refreshments (вүӨ вӮұ1,500),
- **Pay to** вҖ” e.g. "MVP Sports Depot," **Amount**, **What's it for?** вҖ” e.g. "5 jerseys."

The form warns *before* submit if you exceed the category limit
("The contract will reject this") or the balance. Submit вҶ’ the app signs
`request_spend(proposer, category, amount, recipient, memo)`; the proposer's approval is
auto-recorded (**1/2**). Caption: *"Needs 2 of 3 officers to approve before any USDC
moves."*

**Judge-friendly aside:** deliberately request an over-limit Equipment spend (say
вӮұ6,000). The contract rejects it with `Error #3 OverCategoryLimit` вҖ” the policy is
enforced on-chain, not in the UI.

#### Step 4 вҖ” Approve to threshold (Spend tab)

The pending request shows a **`1/2 approvals`** badge and one **approval chip per
officer**. Tap a *second* officer's chip (e.g. Aling Nena) вҶ’ that officer signs
`approve(officer, spend_id)` client-side вҶ’ badge flips to **`2/2`** (green). Tapping the
same officer twice is blocked (`Error #7 AlreadyApproved`). **This is the moat.**

#### Step 5 вҖ” Execute / release (Spend tab)

Only at threshold does the button read **"Release вӮұN in USDC"** and enable. Tap it вҶ’ the
app calls `execute(spend_id)` (permissionless; anyone can push it, but it reverts below
threshold with `Error #6 NotEnoughApprovals`). The contract transfers real USDC to the
recipient and marks the spend executed. The completed row shows **"view tx"** linking to
`stellar.expert`. Open it вҖ” real on-chain proof that money only moved after 2-of-3.

#### Step 6 вҖ” Ask the AI treasurer (Pool tab)

Back on **Pool**, use the **"Ask your AI treasurer"** box. Suggested prompts:
- *"How much do we have right now?"*
- *"Where did the money go last week?"*
- *"Can we afford вӮұ4,000 for new jerseys?"*

The app posts `{question, state}` to `POST /ask`; the model answers in warm, plain
pesos (2вҖ“4 sentences) **grounded only in the on-chain state passed in** вҖ” balance,
contributions, spend history, policy. Close the loop: *"any member can ask, and the
answer is the chain, not a notebook."*

#### Bonus вҖ” Rules вҶ’ policy (Rules tab)

`routes/Setup.tsx` shows the AI turning plain language into policy. Example text is
pre-filled: *"вӮұ200 per member every month. Equipment up to вӮұ5,000, venue up to вӮұ3,000,
snacks up to вӮұ1,500. Any spend over вӮұ5,000 needs 2 of 3 officers to approve."* Tap
**"Parse with AI"** вҶ’ `POST /rules` returns a validated policy (dues, category caps,
2-of-3 approval), rendered as badges. The **Officers** list shows the three on-chain
signers. **"Start over"** clears the browser pool + personas to reset the demo.

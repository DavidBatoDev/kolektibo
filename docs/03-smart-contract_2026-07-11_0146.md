# Smart Contract Reference (Soroban Treasury)
_Created: 2026-07-11 01:46 MPST · Kolektibo · Stellar Testnet_

The treasury contract is the **trust core** of Kolektibo. It is the one piece of the system that holds the money, and it is deliberately the smallest, dumbest, and most auditable piece. The AI treasurer does all the labor (tracking, drafting, answering questions), but it has **zero authority over funds** — every peso lives inside this contract and only moves when the group's own on-chain policy is satisfied. This document is the complete, verified reference for that contract.

Source of truth: `contracts/treasury/src/lib.rs` (306 lines), tests in `contracts/treasury/src/test.rs`, build config in `contracts/Cargo.toml` and `contracts/treasury/Cargo.toml`, live deployment in `DEPLOYMENT.md`.

---

## 1. Purpose

> One instance of this contract = one group's pooled fund.

The contract holds the group's USDC and will **only** release it when the group's own on-chain policy is satisfied:

1. the spend is **within its category limit**, AND
2. **enough officers have approved** (the M-of-N threshold).

No single officer — and no off-chain AI — can move the money. This is the moat: the trust guarantee is enforced by bytecode on Stellar, not by an honest treasurer or a well-behaved model.

```
        ┌─────────────────────────────────────────────┐
        │              GROUP'S POOLED USDC             │
        │        (held by the treasury contract)       │
        └─────────────────────────────────────────────┘
                 ▲                          │
     contribute()│                          │ execute()
       (members) │                          ▼  only if:
                 │              ┌──────────────────────────┐
   ┌─────────────┴───────┐      │ • within category limit  │
   │ AI treasurer / app  │      │ • approvals ≥ threshold  │
   │ (drafts, tracks,    │─────▶│ • not already executed   │
   │  answers — NO keys) │ req  │ • sufficient balance     │
   └─────────────────────┘      └──────────────────────────┘
```

The AI and the app can *propose* and *read*, but the release path (`execute`) is gated entirely by state the contract itself owns.

## 3. Storage keys — `DataKey`

All persisted state is addressed through a single `#[contracttype]` enum. Two of the variants are *parameterized* keys (they carry a value that selects the row).

```rust
#[contracttype]
#[derive(Clone)]
pub enum DataKey {
    Token,
    Officers,
    Threshold,
    NextSpendId,
    Members,
    Categories,
    CategoryLimit(Symbol),
    Contribution(Address),
    Spend(u32),
}
```

| Key | Stored type | Storage layer | Meaning |
|---|---|---|---|
| `Token` | `Address` | instance | The pooled asset (USDC SAC address). Its presence also marks the contract as initialized. |
| `Officers` | `Vec<Address>` | instance | The N officers who may propose/approve spends. |
| `Threshold` | `u32` | instance | M — approvals required to release a spend. |
| `NextSpendId` | `u32` | instance | Monotonic counter; the id the next `request_spend` will use. Starts at `1`. |
| `Members` | `Vec<Address>` | instance | Everyone who has ever contributed (dedup'd). |
| `Categories` | `Vec<Symbol>` | instance | Ordered list of spend category names. |
| `CategoryLimit(Symbol)` | `i128` | instance | Per-spend cap for one category (`0` = no limit). |
| `Contribution(Address)` | `i128` | **persistent** | Running total a given address has contributed. |
| `Spend(u32)` | `SpendRequest` | **persistent** | One spend request by id. |

**Storage-layer note.** Configuration and small aggregates (`Token`, `Officers`, `Threshold`, `NextSpendId`, `Members`, `Categories`, `CategoryLimit`) live in **instance** storage; per-user and per-spend rows (`Contribution`, `Spend`) live in **persistent** storage. `initialize` calls `store.extend_ttl(100_000, 100_000)` on the instance to keep the config alive.

## 4. Structs

### `CategoryInfo`

Returned by the `get_categories` view; feeds the budgets UI and the AI.

```rust
#[contracttype]
#[derive(Clone)]
pub struct CategoryInfo {
    pub name: Symbol,   // e.g. Equipment, Venue, Refreshments
    pub limit: i128,    // per-spend cap in raw units; 0 = no limit
}
```

### `SpendRequest`

The full record of a proposed disbursement and the approvals it has gathered.

```rust
#[contracttype]
#[derive(Clone)]
pub struct SpendRequest {
    pub id: u32,
    pub proposer: Address,
    pub category: Symbol,
    pub amount: i128,          // raw units
    pub recipient: Address,
    pub memo: String,
    pub approvals: Vec<Address>,
    pub executed: bool,
}
```

Field notes:

- `approvals` is a de-duplicated list of officer addresses. The **proposer is auto-added** at creation, so a fresh request already has one approval.
- `executed` flips to `true` exactly once, inside `execute`, after the transfer succeeds. It is the idempotency guard against double-spend.
- `memo` is a human string (e.g. `"2 game balls"`), stored on-chain for the audit trail.

### 5.2 Function prose

#### `initialize(token, officers, threshold, categories, limits)`

One-time setup. Validates before writing anything:

- If `DataKey::Token` already exists → `AlreadyInitialized` (#1).
- If `threshold == 0` **or** `threshold > officers.len()` **or** `categories.len() != limits.len()` → `InvalidInit` (#8).

On success it stores `token`, `officers`, `threshold`, sets `NextSpendId = 1`, initializes `Members` to an empty vector, stores the `categories` list, and writes one `CategoryLimit(categories[i]) = limits[i]` row per category. Finally it extends the instance TTL by `100_000` ledgers. `limits[i] == 0` means "no cap" for that category.

The live testnet policy: **2-of-3** officers, categories **Equipment / Venue / Refreshments** with caps **5,000 / 3,000 / 1,500 USDC** (scaled ×1e7).

#### `contribute(from, amount)`

`from.require_auth()` — the contributor must sign. Rejects `amount <= 0` with `NonPositiveAmount` (#10). Then it pulls the tokens **into** the treasury via the SAC's `transfer(from → contract, amount)`, adds `amount` to `Contribution(from)` (persistent, defaulting a first-time contributor from `0`), and appends `from` to `Members` if not already present. Publishes a `contrib` event. Anyone can contribute — contributors are not required to be officers or pre-registered members.

#### `request_spend(proposer, category, amount, recipient, memo) → u32`

`proposer.require_auth()` then `require_officer(proposer)` — a non-officer is rejected with `NotOfficer` (#2). Rejects `amount <= 0` with `NonPositiveAmount` (#10). Looks up `CategoryLimit(category)`; if a limit exists and is **positive** and `amount > limit`, reverts with `OverCategoryLimit` (#3). (An unknown category has no limit row, so it is treated as uncapped.) It then consumes `NextSpendId` (and increments it), builds a `SpendRequest` with `approvals` pre-seeded with the proposer, persists it under `Spend(id)`, publishes a `spend_req` event carrying the id, and returns the new `id`.

#### `approve(officer, spend_id)`

`officer.require_auth()` then `require_officer(officer)`. Loads `Spend(spend_id)` or reverts `SpendNotFound` (#4). Reverts `AlreadyExecuted` (#5) if the spend is already done, and `AlreadyApproved` (#7) if this officer's address is already in `approvals`. Otherwise appends the officer to `approvals`, re-persists, and publishes an `approve` event `(spend_id, officer)`.

#### `execute(spend_id)`

**Permissionless** — there is no `require_auth`; anyone in the group can push the button. This is safe precisely because the release conditions are checked against contract-owned state:

1. Load `Spend(spend_id)` or revert `SpendNotFound` (#4).
2. Revert `AlreadyExecuted` (#5) if already executed — the double-spend guard.
3. Load `Threshold`; if `approvals.len() < threshold` → `NotEnoughApprovals` (#6).
4. Check the treasury's live token balance; if `< amount` → `InsufficientBalance` (#9).
5. `transfer(contract → recipient, amount)`, set `executed = true`, persist, publish an `execute` event `(spend_id, amount)`.

Making `execute` permissionless is intentional UX: once officers have signed off, *any* member (or the app) can trigger the release, and the contract still refuses if the policy isn't met. Authority lives in the approvals, not in who presses the button.

#### Read-only views

`get_balance` returns the treasury's live token balance (queried from the SAC, not a cached number). `get_spend(id)` returns `Option<SpendRequest>`. `get_spends()` iterates `1..NextSpendId` and returns every request (pending + executed), ids ascending. `get_categories()` returns `Vec<CategoryInfo>` for the budgets UI. The rest (`get_officers`, `get_threshold`, `get_members`, `get_contribution(who)`, `get_next_spend_id`) expose config and aggregates. `get_contribution` defaults to `0` for an unknown address.

#### `require_officer` (private helper)

```rust
fn require_officer(env: &Env, who: &Address) {
    let officers: Vec<Address> = env.storage().instance().get(&DataKey::Officers).unwrap();
    if !officers.contains(who) {
        panic_with_error!(env, Error::NotOfficer);
    }
}
```

Called by both `request_spend` and `approve`. Note it runs **after** `require_auth`, so an attacker cannot even reach the officer check without a valid signature.

## 6. Errors — `Error` enum

Declared with `#[contracterror]`, `#[repr(u32)]`. Every revert path uses `panic_with_error!`, which surfaces on-chain as `Error(Contract, #N)`.

| # | Variant | Raised by | Meaning |
|---|---|---|---|
| 1 | `AlreadyInitialized` | `initialize` | Contract already set up; refuses re-initialization. |
| 2 | `NotOfficer` | `request_spend`, `approve` | Caller is not in the officers list. |
| 3 | `OverCategoryLimit` | `request_spend` | Requested amount exceeds the category's positive cap. |
| 4 | `SpendNotFound` | `approve`, `execute` | No `SpendRequest` exists for that id. |
| 5 | `AlreadyExecuted` | `approve`, `execute` | Spend already released; blocks double-spend. |
| 6 | `NotEnoughApprovals` | `execute` | Fewer approvals than the threshold — the core "one person can't drain it" guard. |
| 7 | `AlreadyApproved` | `approve` | This officer has already approved this spend. |
| 8 | `InvalidInit` | `initialize` | Bad init args: threshold `0`, threshold `>` officer count, or `categories.len() != limits.len()`. |
| 9 | `InsufficientBalance` | `execute` | Treasury balance is less than the spend amount. |
| 10 | `NonPositiveAmount` | `contribute`, `request_spend` | Amount is `<= 0`. |

```rust
#[contracterror]
#[derive(Copy, Clone, Debug, Eq, PartialEq, PartialOrd, Ord)]
#[repr(u32)]
pub enum Error {
    AlreadyInitialized = 1,
    NotOfficer         = 2,
    OverCategoryLimit  = 3,
    SpendNotFound      = 4,
    AlreadyExecuted    = 5,
    NotEnoughApprovals = 6,
    AlreadyApproved    = 7,
    InvalidInit        = 8,
    InsufficientBalance= 9,
    NonPositiveAmount  = 10,
}
```

## 8. Units convention

The contract is **unit-agnostic**: every amount is a raw `i128`. Meaning is imposed by the token's decimals plus one app-wide convention.

- The USDC Stellar Asset Contract has **7 decimals**, so `1 USDC = 10,000,000 raw`.
- App/deploy convention: **`raw = human × 10_000_000`** (`SCALE = 1e7`).
- For the demo, **1 USDC ≈ ₱1**, so the group thinks in pesos while settlement is USDC on-chain.

```
raw = humanAmount * 10_000_000        // SCALE = 1e7
```

| Human | Raw `i128` |
|---|---|
| ₱200 dues | `2_000_000_000` |
| ₱5,000 Equipment cap | `50_000_000_000` |
| 20,000 USDC treasury seed | `200_000_000_000` |

The frontend formats raw → ₱ for display and multiplies by `SCALE` for any contract call. `i128` gives ~1.7×10³⁸ headroom, so overflow is not a practical concern even before `overflow-checks` catches it.

## 9. Trust guarantees

These are the properties the contract *structurally* enforces — the reason the fund is trustworthy without trusting any one person or the AI.

1. **No single officer can release funds.** `execute` reverts with `NotEnoughApprovals` (#6) until `approvals ≥ threshold`. With a 2-of-3 policy, one officer signing is not enough.
2. **The AI has no authority over money.** The AI holds no keys and calls no write path with signing power beyond what a normal client can do; `execute` only ever moves funds that already satisfy on-chain policy. The AI can draft a spend, never release one.
3. **Only officers can propose or approve.** Enforced by `require_officer` after `require_auth` in both `request_spend` and `approve` (`NotOfficer`, #2).
4. **Category limits are hard caps.** An over-limit spend is rejected at proposal time (`OverCategoryLimit`, #3) — it never even becomes a pending request.
5. **No double-spend.** The `executed` flag plus the `AlreadyExecuted` (#5) guard in both `approve` and `execute` make release idempotent.
6. **No duplicate approvals.** `AlreadyApproved` (#7) stops one officer from padding the count toward the threshold.
7. **Can't overdraw.** `execute` checks the live balance and reverts `InsufficientBalance` (#9) rather than attempting a failing transfer.
8. **Immutable policy after setup.** `initialize` is one-time (`AlreadyInitialized`, #1) with validated inputs (`InvalidInit`, #8); the officer set, threshold, and category caps cannot be silently re-pointed.

### Proven on-chain (from `DEPLOYMENT.md`)

Policy: 2-of-3 officers, Equipment/Venue/Refreshments caps, treasury seeded with 20,000 test USDC.

```
1. request_spend        → spend #1, proposer auto-approved (1 approval)
2. execute (1 approval) → REVERT Error(Contract, #6) NotEnoughApprovals   ← can't drain w/ one officer
3. approve (2nd officer)→ 2 approvals;  duplicate approve → REVERT #7 AlreadyApproved
4. execute (2-of-3)     → real USDC transfer event, treasury → recipient (1000 raw)
5. balances on-chain:   treasury 200000000000 → 199999999000, recipient 0 → 1000
```

## 10. Test suite

Location: `contracts/treasury/src/test.rs`. **4 tests, 4 passing** (`cargo test`). A shared `setup()` builds an `Env` with `mock_all_auths()`, registers a v2 Stellar Asset Contract as test USDC, mints `10_000` raw to a member, deploys the treasury, and initializes it **2-of-3** with categories `Equipment` (cap `5_000`) and `Venue` (cap `3_000`).

> Test amounts use small raw integers (`2_000`, `1_200`, `5_000`) for readability; they are not ×1e7-scaled. The scaling convention is an app/deploy concern, not a contract concern — the contract is unit-agnostic.

| Test | What it exercises | What it proves | Asserts |
|---|---|---|---|
| `full_happy_path` | contribute → request_spend (auto-approve) → approve → execute | The complete legitimate loop works; funds land with the vendor and balances update. | vendor balance `1_200`, treasury `800`, `get_spend(id).executed == true` |
| `cannot_execute_without_threshold` | request_spend, then `try_execute` with only the proposer's 1 approval | A spend cannot be released below threshold — the core drain-protection. | `Err(Ok(Error::from_contract_error(NotEnoughApprovals=6)))`; vendor balance still `0` |
| `rejects_over_category_limit` | `try_request_spend` for `6_000` against the `5_000` Equipment cap | Over-limit spends are rejected at proposal time. | `Err(Ok(Error::from_contract_error(OverCategoryLimit=3)))` |
| `non_officer_cannot_propose` | `try_request_spend` signed by a plain member | Non-officers cannot even create a spend request. | `Err(Ok(Error::from_contract_error(NotOfficer=2)))` |

The three negative tests use the generated `try_*` client methods, which return a `Result` instead of panicking, and assert against `soroban_sdk::Error::from_contract_error(code)`. This is because a `panic_with_error!` surfaces through the `try_` client as a `soroban_sdk::Error`, not as the raw `Error` enum — so tests compare against `from_contract_error(Error::Variant as u32)`.

Example (negative-path assertion shape):

```rust
let res = s.client.try_execute(&id);
assert_eq!(
    res,
    Err(Ok(soroban_sdk::Error::from_contract_error(
        Error::NotEnoughApprovals as u32
    )))
);
```

**Trustline caveat (from deploy).** Classic Stellar assets (including real Circle USDC) require a recipient account to establish a **trustline** before it can receive the asset. Contracts are exempt — which is why minting into the treasury worked, but the first transfer to a fresh account failed until a `change-trust` was added. App onboarding creates trustlines client-side for members and payees before disbursement; a real SEP-24 anchor would do this automatically.

## 12. At a glance

```
initialize ─(one-time, validated)─▶ [ Token, Officers, Threshold=M,
                                       NextSpendId=1, Members=[], Categories, Limits ]

contribute(from,amt)  ─auth(from)──▶ pull tokens in, +Contribution, +Members, emit contrib
request_spend(...)    ─auth+officer▶ check cap, new SpendRequest{approvals:[proposer]}, emit spend_req
approve(officer,id)   ─auth+officer▶ +approval (no dup, not executed), emit approve
execute(id)           ─permissionless▶ require approvals≥M & balance≥amt, transfer out, emit execute
```

The contract is intentionally minimal and fully covered: **~300 lines, 10 error codes, 4 events, 4 passing tests, verified end-to-end on Stellar testnet.** The trust guarantee is not a promise — it is the bytecode.

# Frontend (Web PWA) Reference
_Created: 2026-07-11 01:46 MPST · Kolektibo · Stellar Testnet_

The `apps/web` package is Kolektibo's user-facing app: a mobile-first, installable Progressive Web App that lets an ordinary Filipino group create a treasury, contribute real USDC, propose spends, collect on-chain approvals, release funds, and ask an AI treasurer plain-language questions — all while the money lives in a Soroban smart contract that no single person (and not the AI) can drain.

This reference documents the stack, every library module, the generated contract bindings, each screen and component, the create-your-pool onboarding flow, the pesos-vs-USDC display model, and the smoke tests. All names, signatures, versions, and IDs below are verified against the repository source.

### Buffer polyfill + tsconfig notes

Two browser/toolchain gotchas are solved in config, not code:

1. **Buffer polyfill.** `@stellar/stellar-sdk` (and the generated bindings) expect Node's `Buffer`, `global`, and `process`. `vite-plugin-node-polyfills` injects them. The generated bindings additionally self-guard at module load:
   ```ts
   if (typeof window !== "undefined") { window.Buffer = window.Buffer || Buffer; }
   ```
2. **tsconfig for generated bindings.** The Stellar contract bindings are emitted with type-only imports that clash with strict settings, so `apps/web/tsconfig.app.json` sets:
   - `"verbatimModuleSyntax": false` — lets mixed value/type imports from the bindings compile.
   - `"types": ["vite/client", "vite-plugin-pwa/client", "node"]` — pulls in `import.meta.env`, the PWA client types, and Node types the bindings reference.
   - `"strict": true`, `"moduleResolution": "bundler"`, `"allowImportingTsExtensions": true`, `"jsx": "react-jsx"`, `target/module` `ES2022`/`ESNext`.

   `tsconfig.json` is a solution file referencing `tsconfig.app.json` (app source under `src`) and `tsconfig.node.json` (`vite.config.ts` only).

## 3. Library modules (`src/lib`)

### 3.1 `wallet.ts` — officer personas (in-browser keypairs)

Kolektibo manages the group's three officers as **in-browser testnet keypairs** persisted in localStorage under `kolektibo.personas.v1`. Secrets are generated on-device and never leave it — the app signs each officer's approval locally. In production these are three separate people's wallets.

```ts
export type Persona = { name: string; publicKey: string; secret: string }
export const OFFICER_NAMES = ['Kap. Ramon', 'Aling Nena', 'Kuya Jun'] as const
```

| Function | Signature | Behavior |
| --- | --- | --- |
| `getPersonas` | `(): Persona[]` | Returns the 3 stored personas; if missing/corrupt, generates 3 fresh `Keypair.random()` personas named by `OFFICER_NAMES` and persists them |
| `personaByName` | `(name: string): Persona \| undefined` | Lookup by display name |
| `keypairFor` | `(p: Persona): Keypair` | `Keypair.fromSecret(p.secret)` — rehydrates a signer |
| `resetPersonas` | `(): void` | Removes the localStorage key (used by "Start over") |

### 3.2 `contract.ts` — read client, unit conversion, active pool

Wraps the generated `Client` for read-only view calls and holds the pesos↔raw conversion.

```ts
export const TREASURY_ID =
  import.meta.env.VITE_TREASURY_CONTRACT_ID || networks.testnet.contractId
export const SCALE = 10_000_000                              // USDC SAC = 7 decimals
export const rawToUsd = (raw: bigint): number => Number(raw) / SCALE
export const usdToRaw = (n: number): bigint => BigInt(Math.round(n * SCALE))
```

| Export | Signature | Notes |
| --- | --- | --- |
| `TREASURY_ID` | `string` | Env override, else the canonical demo pool baked into the bindings (`CBR36Q2A…KINF2`) |
| `activePoolId` | `(): string` | Returns the browser-created pool id (`localStorage['kolektibo.contract']`) **else** `TREASURY_ID` |
| `readClient` | `(contractId = activePoolId()): Client` | No-signer `Client` for simulated (read) calls |
| `fetchLiveTreasury` | `(): Promise<LiveTreasury>` | Parallel `get_balance/threshold/officers/categories/members/spends`, returns raw shapes |
| `contractExplorerUrl` | `(id = TREASURY_ID): string` | `https://stellar.expert/explorer/testnet/contract/<id>` |

`LiveTreasury` type: `{ contractId, balance: bigint, threshold: number, officers: string[], categories: CategoryInfo[], members: string[], spends: SpendRequest[] }`.

### 3.3 `backend.ts` — thin chain-ops backend client

Talks to the AI/chain-ops backend at `import.meta.env.VITE_AI_URL || 'http://localhost:8787'`.

| Function | Signature | Endpoint | Notes |
| --- | --- | --- | --- |
| `getConfig` | `(): Promise<ChainConfig>` | `GET /config` | Result is **cached** in a module-level `cached` variable |
| `faucet` | `(address: string): Promise<void>` | `POST /faucet` | Mints test USDC to an address (must already trust USDC) |
| `createPoolOnChain` | `(officers: string[], threshold = 2): Promise<string>` | `POST /pool/create` | Deploys + initializes a fresh treasury; returns `contractId` |

`ChainConfig` fields: `{ network, usdcSac, usdcIssuer, rpcUrl, passphrase, friendbotUrl, categories: string[], limits: string[], threshold: number, configured: boolean }`. Errors surface the response body text when present (`throw new Error(await res.text())`).

### 3.4 `stellar.ts` — network constants + Horizon helpers

```ts
export const NETWORK = {
  passphrase: Networks.TESTNET,
  horizonUrl:    import.meta.env.VITE_HORIZON_URL      || 'https://horizon-testnet.stellar.org',
  sorobanRpcUrl: import.meta.env.VITE_SOROBAN_RPC_URL  || 'https://soroban-testnet.stellar.org',
  friendbotUrl:  'https://friendbot.stellar.org',
}
export const horizon = new Horizon.Server(NETWORK.horizonUrl)
export const soroban = new rpc.Server(NETWORK.sorobanRpcUrl)
```

Helpers: `fundWithFriendbot(pk)` (tolerates HTTP 400 = "already funded"), `getAccountSummary(pk)` (`{ exists, xlm, balances }`, treats 404/`NotFoundError` as not-funded), `explorerAccountUrl(pk)`, `explorerTxUrl(hash)`.

### 3.5 `identity.ts` — single demo wallet

A separate single testnet keypair in `localStorage['kolektibo.secret']` used by the (currently unmounted) `WalletCard`: `getIdentity(): Keypair`, `resetIdentity()`, and `shortAddr(pk, head=5, tail=4)` for address truncation used across screens.

### 3.6 `ai.ts` — AI treasurer client

Talks to the same backend base URL. Two calls:

```ts
export type Policy = {
  currency: string
  dues: { amount: number; period: 'monthly'|'weekly'|'once' } | null
  categories: { name: string; monthlyLimit: number | null }[]
  approval: { threshold: number; of: number }
  summary: string
}

askAI(question: string, state: unknown): Promise<string>  // POST /ask → { answer }
```

### 3.7 `pool.ts` — view-model types + math

Defines the app's view model (`Member`, `Officer`, `Spend`, `Pool`) and the on-chain-mirroring shapes. The live path uses only:

- `poolBalance(pool): number` = Σ member contributions − Σ executed spend amounts (the app-side derived balance shown in the hero/Spend cards).
- The `Spend`/`Pool`/`Member`/`Officer` type definitions that `livepool.readPool()` populates from chain data.

The file also retains an older localStorage "DEMO STORE" (`seed()`, `loadPool`, `savePool`, `ensurePool`, `resetPool`, `clearPool`) documenting the Day-1 mock. The live app no longer reads from it — real state comes from `livepool.ts`.

### 3.8 `livepool.ts` — the live on-chain path (the core module)

This is where the browser signs and submits to Stellar. It uses the generated `Client` plus `basicNodeSigner` from `@stellar/stellar-sdk/contract`.

**localStorage keys:** `kolektibo.contract` (`POOL_KEY`, the deployed pool id) and `kolektibo.payee.v1` (`PAYEE_KEY`, a generated recipient account for demo spends).

**Pool id helpers:** `getPoolId(): string | null`, `setPoolId(id)` (private), `clearPool(): void`.

**The `client()` factory** builds a bindings `Client` pointed at testnet; with a keypair it attaches client-side signing:

```ts
function client(contractId: string, signerKp?: Keypair): Client {
  const base = { contractId, networkPassphrase: NETWORK.passphrase, rpcUrl: NETWORK.sorobanRpcUrl }
  if (!signerKp) return new Client(base)                       // read-only
  const signer = basicNodeSigner(signerKp, NETWORK.passphrase) // sign locally
  return new Client({ ...base, publicKey: signerKp.publicKey(),
                      signTransaction: signer.signTransaction,
                      signAuthEntry:   signer.signAuthEntry })
}
```

**Account plumbing (private):**

- `ensureFunded(pk)` — `horizon.loadAccount(pk)`; on failure calls `fundWithFriendbot(pk)`.
- `ensureTrustline(kp, issuer)` — checks the account's balances for a `USDC` trustline to `issuer`; if absent, builds/signs/submits a `changeTrust` op (`Asset('USDC', issuer)`, `BASE_FEE`, `setTimeout(60)`). This is required because a classic asset (SAC) recipient **cannot receive USDC until it trusts USDC** — the fix for the "first transfer to a fresh account fails" problem.

**`createPool(onProgress)`** — see §5 for the step-by-step flow.

**Chain-signed writes** (each resolves the persona → keypair → signed `AssembledTransaction` → `signAndSend()`):

| Function | Signature | Contract call |
| --- | --- | --- |
| `contribute` | `(personaName, amountUsd): Promise<void>` | `contribute({ from, amount: usdToRaw(amountUsd) })` |
| `requestSpend` | `(officerName, category, amountUsd, memo): Promise<number>` | `request_spend({ proposer, category, amount, recipient: getPayee().publicKey, memo })`; returns `Number(at.result)` (the new spend id) |
| `approve` | `(officerName, spendId): Promise<void>` | `approve({ officer, spend_id: spendId })` |
| `execute` | `(spendId): Promise<string>` | `execute({ spend_id })` signed by `getPersonas()[0]` (permissionless — anyone can push the button); returns the tx hash (`sent.sendTransactionResponse?.hash ?? ''`) |

**`readPool(): Promise<Pool | null>`** — the read → view-model transform:

1. Reads six views in parallel: `get_balance`, `get_threshold`, `get_officers`, `get_categories`, `get_members`, `get_spends`.
2. Then reads each member's `get_contribution({ who })` in parallel.
3. Maps public keys → persona display names (`nameFor`, falling back to a `4…4` short address).
4. Converts every raw `i128` amount to pesos via `rawToUsd`.
5. Returns a `Pool` with a hard-coded display name **"Barangay 143 Basketball League"**, `currency: 'USDC'`, a synthesized `policy` (`dues 200/monthly`, category caps from chain, `approval { threshold, of: officers.length }`, and a summary string `"Any spend needs {threshold} of {n} officers to approve. Enforced on-chain."`), officers, members (with contributed pesos), and spends. (`createdAt` is `0` and `recipientName` is a short address, since those aren't stored on-chain.)

## 4. Generated bindings (`src/contract/treasury/src/index.ts`)

Produced by `stellar contract bindings typescript` from the deployed WASM. It re-exports the whole SDK (`export * from "@stellar/stellar-sdk"`) plus `contract` and `rpc` namespaces, and self-installs the Buffer shim on `window`.

**`networks`** pins the canonical demo pool:

```ts
export const networks = {
  testnet: {
    networkPassphrase: "Test SDF Network ; September 2015",
    contractId: "CBR36Q2AEAUQWZ6CXESIYEGWPYCDUDHQP62EEYFHS5JELW4T3FGKINF2",
  }
} as const
```

**`Errors`** maps the contract's `u32` codes to messages: `1 AlreadyInitialized, 2 NotOfficer, 3 OverCategoryLimit, 4 SpendNotFound, 5 AlreadyExecuted, 6 NotEnoughApprovals, 7 AlreadyApproved, 8 InvalidInit, 9 InsufficientBalance, 10 NonPositiveAmount`.

**Types:** `DataKey` (storage-key union), `CategoryInfo { limit: i128; name: string }`, `SpendRequest { id: u32; proposer; category; amount: i128; recipient; memo; approvals: Array<string>; executed: boolean }`.

**`Client` methods** (each returns `Promise<AssembledTransaction<T>>`; call `.signAndSend()` for state-changing ones):

| Method | Args | Result `T` |
| --- | --- | --- |
| `initialize` | `{ token, officers[], threshold, categories[], limits[] }` | `null` |
| `contribute` | `{ from, amount }` | `null` |
| `request_spend` | `{ proposer, category, amount, recipient, memo }` | `u32` |
| `approve` | `{ officer, spend_id }` | `null` |
| `execute` | `{ spend_id }` | `null` |
| `get_balance` | — | `i128` |
| `get_threshold` | — | `u32` |
| `get_officers` / `get_members` | — | `Array<string>` |
| `get_categories` | — | `Array<CategoryInfo>` |
| `get_spends` | — | `Array<SpendRequest>` |
| `get_spend` | `{ id }` | `Option<SpendRequest>` |
| `get_contribution` | `{ who }` | `i128` |
| `get_next_spend_id` | — | `u32` |

The `Client` constructor embeds the contract's XDR `ContractSpec` (base64 spec entries) so simulation/decoding works without a network round-trip for the interface. `Client.deploy(...)` is available for programmatic deploys (the app deploys via the backend instead).

## 5. Create-your-pool onboarding flow

Triggered by `CreatePool` → `useCreatePool()` → `livepool.createPool(onProgress)`. It provisions a real, fresh treasury per group and seeds it with real on-chain contributions. Progress strings stream to the UI.

```
createPool(onProgress):
 0. cfg = getConfig(); throw if !cfg.configured        "Backend chain config missing"
 1. personas = getPersonas() (3)     ; payee = getPayee()
 2. for each persona:                                   "Setting up {name}…"
      ensureFunded(pk)         → Friendbot creates/funds the account (XLM)
      ensureTrustline(pk, cfg.usdcIssuer) → changeTrust USDC (so it can hold USDC)
 3. payee: ensureFunded + ensureTrustline               "Setting up payee account…"
 4. contractId = createPoolOnChain(                     "Deploying your treasury on Stellar…"
        personas.map(p => p.publicKey), cfg.threshold)  → POST /pool/create (deploy+initialize)
    setPoolId(contractId)      → localStorage['kolektibo.contract']
 5. for each persona: faucet(pk)                        "Funding {name} with test USDC…"
 6. seed 3 real contributions (chain-signed):           "{name} contributing ₱{amt}…"
        Kap. Ramon 600, Aling Nena 400, Kuya Jun 600
 7. onProgress('Done'); return contractId
```

On success the hook invalidates the `['pool']` query, so Home re-reads live state and swaps the onboarding card for the dashboard. Total inflow seeded = ₱1,600 (600+400+600).

**Why per-pool deploy:** each group gets its own treasury contract instance with the three browser personas as officers, initialized to 2-of-3 with the backend's categories/limits (Equipment/Venue/Refreshments). The canonical pool `CBR36Q2A…KINF2` remains the read-only fallback (`activePoolId()`) when a browser hasn't created its own.

## 6. Hooks (`src/hooks/usePool.ts`)

All query/mutation state flows through one query key: `POOL_KEY = ['pool']`.

| Hook | Returns | Behavior |
| --- | --- | --- |
| `usePool()` | `UseQueryResult<Pool \| null>` | `queryFn: live.readPool`, `staleTime: 5_000`, `retry: 1` |
| `useHasPool()` | `boolean` | `Boolean(live.getPoolId())` — has this browser deployed a pool? |
| `usePoolBalance()` | `number` | `poolBalance(data)` (pesos) or `0` while loading |
| `useCreatePool()` | `{ ...mutation, progress }` | Wraps `live.createPool(setProgress)`; `onSuccess` invalidates `['pool']`; exposes the streaming `progress` string |
| `usePoolActions()` | `{ contribute, requestSpend, approveSpend, executeSpend }` | Four mutations; every `onSuccess` invalidates `['pool']` so the UI re-reads chain state |

`usePoolActions` mutation input shapes: `contribute {personaName, amount}`, `requestSpend {officerName, category, amount, memo}`, `approveSpend {id, officerName}`, `executeSpend {id}`.

`LiveOnChain` uses its own separate query key `['live-treasury']` (see §7.2).

### 7.1 `AppShell` (`components/AppShell.tsx`) — root layout

A centered `max-w-md` phone frame (`min-h-dvh`, dark) with:

- **Sticky header:** `kolektibo.svg` logo + "Kolektibo" wordmark + a "Stellar Testnet" pill.

### 7.2 `LiveOnChain` (`components/LiveOnChain.tsx`) — live proof card

The trust centerpiece. `useQuery({ queryKey: ['live-treasury'], queryFn: fetchLiveTreasury, refetchInterval: 15_000 })` re-reads the contract every 15s. Renders:

- Contract **balance** in USDC (`rawToUsd`, `toLocaleString`) + a "N of M to approve" gold badge.
- **Officers** as chips: `OFFICER_NAMES[i]` (fallback `Officer i+1`) + `shortAddr(addr,4,4)`, full key in `title`. (Order = deploy order: deployer, officer2, officer3.)
- **Category caps** "enforced by contract" (name → `≤ limit USDC`).
- A monospace link to the contract on stellar.expert.
- A pulsing green "live · testnet" indicator; loading/error states ("Reading Stellar…" / "Couldn't reach Soroban RPC. Retrying…").

### 7.3 `CreatePool` (`components/CreatePool.tsx`) — onboarding CTA

Card explaining "This deploys a fresh treasury with 3 officers (2-of-3 approval), gives each test USDC, and seeds the pool with real on-chain contributions. Takes about a minute." The button calls `create.mutate()`; while pending it shows the live `create.progress` string (§5) and an error line if the deploy fails.

- **Pool loading:** "Reading your pool from Stellar…".
- **Dashboard:**
  1. **Balance hero** — `peso(balance)`, pool name, "settles in USDC on Stellar", member count, and the policy summary rule.

  3. **`AskTreasurer`** — an inline component posting to `askAI(question, { pool, currency, balance, policy, members, spends })`. ### 7.5 Contribute (`routes/Contribute.tsx`)

- Persona picker (`getPersonas()`), amount field (default = `pool.policy.dues.amount` or 200), quick-amount chips ₱100/₱200/₱500.
- **Contribute button** → `contribute.mutate({ personaName, amount })` → chain-signed `contribute`. Status lines: "Signing & submitting to Stellar…", "Contribution confirmed on-chain ✓", or the error message.

### 7.6 Spend (`routes/Spend.tsx`) — the moat, demonstrated

- **Header card:** "Available to spend" `peso(balance)` + a "Request a spend" toggle.
- **Request form:** proposer (officer select), category (with `≤ cap` hint), recipient name, amount, memo. Client-side guards mirror the contract: `overLimit` (amount > category cap → "The contract will reject this") and `overBalance` ("Exceeds the pool balance"); Submit is disabled unless category, amount, and recipient are set and neither guard trips. On submit → `requestSpend.mutate` with `memo = "{recipientName} — {memo}"`. Footer: "Needs {threshold} of {of} officers to approve before any USDC moves."
- **Awaiting approval** → one `SpendRow` per pending spend:
  - Amount → recipient, category · memo · by proposer, and an `x/threshold approvals` badge.
  - **Per-officer approval chips:** each officer is a button; if already in `spend.approvals` it renders `✓ Name` (disabled, green), else `+ Name` → `approveSpend.mutate({ id, officerName })`.
  - **Release button:** enabled only when `approvals.length >= threshold` **and** `amount <= balance` (`canExecute`). Reads "Release {peso} in USDC" when ready, else "Waiting for approvals" → `executeSpend.mutate({ id })`.
- **Completed** list: executed spends with a `view tx`/`demo tx` explorer link (`explorerTxUrl`, or none for `demo`-prefixed hashes).

The 2-of-3 gate and the permissionless release are exactly steps 5–6 of the winning demo loop — enforced on-chain, only surfaced by the UI.

## 8. The pesos-vs-USDC display model

The group thinks in **pesos**; the chain settles in **USDC** (1 USDC ≈ 1 peso for the demo). One convention governs the boundary:

```
raw (on-chain i128, 7-decimal USDC SAC)  =  pesos × 10_000_000
```

| Direction | Function | Where |
| --- | --- | --- |
| pesos → raw | `usdToRaw(n) = BigInt(Math.round(n * SCALE))` | every write (`contribute`, `request_spend`) |
| raw → pesos | `rawToUsd(raw) = Number(raw) / SCALE` | every read (`readPool`, `LiveOnChain`) |

`SCALE = 10_000_000` (`1e7`). Display rules:

- **Peso-facing surfaces** (hero, budgets, Contribute, Spend forms/rows, activity) use `peso()` — the ₱ symbol, no decimals.
- **`LiveOnChain`** deliberately labels amounts in **USDC** (not pesos) to make the on-chain settlement asset explicit for judges — same numbers, different unit label.
- Contract amounts are unit-agnostic `i128`; the 7-decimal scaling is the app's convention, matching the USDC SAC's decimals so raw values line up with real USDC balances (verified by the write smoke test: `300n * SCALE` in the contract ⇔ `300` USDC on the payee's classic balance).

## 9. Smoke tests (`apps/web/scripts`)

Two `.mts` scripts (run with `tsx`) validate the exact primitives the browser uses — the generated bindings and `basicNodeSigner` — against live testnet.

### 9.1 `smoke.mts` — read path

Instantiates a no-signer `Client` on the canonical pool (`networks.testnet.contractId`) and logs `get_balance` (raw), `get_threshold`, `get_officers`, `get_categories`, `get_members`, `get_spends`. Verified result: balance `200000000000n` = 20,000 USDC, threshold 2, three officers, scaled category caps.

### 9.2 `smoke-write.mts` — full write path (E2E)

Mirrors the browser's write architecture end-to-end using `basicNodeSigner`:

```
1) fund 3 random officers + payee via Friendbot; add USDC trustlines
2) POST /pool/create { officers, threshold: 2 }        → fresh contractId
3) POST /faucet for each officer                       → test USDC
4) officer0.contribute(600 × SCALE)                    → signAndSend
5) officer0.request_spend(300 Equipment → payee)       → spend id
6) officer1.approve(spend id)                          → now 2 of 3
7) officer0.execute(spend id)                          → releases USDC
   assert: pool balance === 300 × SCALE  AND  payee USDC === 300  → ✅ PASS
```

`wc(contractId, kp)` builds a signing `Client` exactly like `livepool.client()`. The PASS proves the in-browser-officer-keys + client-side-signing model works on-chain (600 USDC in − 300 out = 300 remaining; payee received 300).
